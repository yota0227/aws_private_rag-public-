# BOS-AI RAG 시스템 현황 및 확장 계획

**보고일:** 2026-04-28
**버전:** v1.2
**대상:** 경영진

---

## 1. 왜 시작했는가

반도체 설계에서 **문서 작성**은 가장 큰 반복 병목이다.

- 블록 HDD 1건 작성에 **2~4주**, 칩 전체로는 **수십 인·월**
- 코드는 바뀌는데 문서가 안 따라감 → 인증(ISO 26262, ASPICE) 직전 병목
- FMEDA, Testbench, Waiver 등 **팀마다 유사한 반복 업무**가 존재

BOS-AI 팀은 이 문제를 **AI 기반 RAG 시스템**으로 해결하고자 RTL 소스 코드 자동 분석 파이프라인을 구축했다.

## 3. 현재 한계와 원인

### 되는 것 vs 안 되는 것

| ✅ 되는 것 | ❌ 안 되는 것 |
|-----------|-------------|
| 모듈명·포트·인스턴스 정확 추출 (17개) | 패키지 상수 (그리드 크기, 타일 타입) |
| 14섹션 HDD 초안 자동 생성 | 알고리즘 동작 설명 |
| 검색 근거 추적 체계 | 전력 관리, 리셋, 주소 맵 |

### 왜 안 되는가

RTL 코드 = **"어떻게 만들었는가(How)"**. 설계 의도 = **Spec 문서에 있다.**

| 정보 출처 | 정답지 기여도 | 현재 커버 |
|----------|-------------|----------|
| **Spec 문서** | 60% | ❌ 없음 |
| **RTL 코드** | 25% | ✅ 현재 |
| **엔지니어 경험** | 15% | ❌ |

**59%는 RTL 단독의 거의 천장이다.** 다음 도약은 Spec 문서를 추가하는 것.

## 6. 인프라와 인력

### 인프라: 이미 있다

RTL RAG 구축 과정에서 만든 인프라(OpenSearch, Bedrock, Lambda, MCP Bridge)를 **그대로 재사용**. 신규 RAG 추가 시 **파서 코드 + 인덱스 정의만** 추가. 인프라 추가 투자 최소.

### 사람: 부족하다

| 현재 | 문제 |
|------|------|
| **1인**이 전부 수행 | Spec RAG 구축과 **동시 진행 불가** |

**요청: RAG 엔지니어 1명 즉시 충원**

| 시나리오 | Phase 1 완료 | Phase 2 시작 |
|----------|-------------|-------------|
| 현행 1인 | 2026년 말 | 2027 Q1 |
| **2인 체제** | **Q3 2026** | **Q4 2026** |

**1명 추가 = 6개월 앞당김.**

### 충원 근거

- 1인 × 12일로 0→59% — **생산성 검증 완료**
- 인프라·방법론 재사용 → 추가 인력 **즉시 생산성 발현**
- RAG Farm = 설계·DV·Safety·FW·품질 **전 팀** 기반 시스템
- 인증 준비 단축 = **프로젝트 일정 리스크 감소**
- 기술 지식 **시스템 축적** → 이직 리스크 완화
