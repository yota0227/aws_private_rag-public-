# BOS-AI RAG 시스템 현황 및 확장 계획

**보고일:** 2026-04-28
**버전:** v1.1
**대상:** 경영진

---

## 1. 왜 시작했는가

반도체 설계 과정에서 **설계 문서(HDD) 작성**은 가장 시간이 많이 드는 반복 업무 중 하나다. 엔지니어 1명이 하나의 블록 HDD를 작성하는 데 **2~4주**가 소요되며, 칩 전체로는 수십 인·월이 투입된다. 코드는 바뀌는데 문서가 안 따라가는 것이 만성적 병목이다.

BOS-AI 팀은 이 문제를 **AI 기반 RAG(Retrieval-Augmented Generation) 시스템**으로 해결하고자 했다. RTL 소스 코드를 자동 분석하여 설계 문서 초안을 생성하는 파이프라인을 구축했다.

## 3. 현재 어디까지 왔는가

### 되는 것

- RTL 코드에서 **17개 모듈의 이름, 포트, 인스턴스 관계**를 정확히 추출
- **14개 섹션**으로 구성된 칩 전체 HDD 초안을 자동 생성
- 검색 결과의 출처를 추적할 수 있는 **투명한 근거 체계** 확보

### 안 되는 것

- 패키지 상수 (그리드 크기, 타일 타입 정의)
- 알고리즘 동작 설명 (라우팅 로직, 주소 변환 절차)
- 전력 관리, 리셋 체인, 주소 맵

### 왜 안 되는가

RTL 코드는 **"어떻게 만들었는가(How)"**만 담고 있다. **"무엇을 만들려 했는가(What)"**는 설계 Spec 문서에 있다.

정답지를 분석한 결과:

| 정보 출처 | 정답지 기여도 | 현재 RAG 커버 |
|----------|-------------|-------------|
| **Spec 문서** (설계 사양서) | 60% | ❌ 아직 없음 |
| **RTL 코드** (구현체) | 25% | ✅ 현재 수준 |
| **엔지니어 경험** | 15% | ❌ |

**현재 59%는 RTL에서 뽑을 수 있는 것의 거의 최대치에 가깝다.** 다음 도약은 Spec 문서를 RAG에 추가하는 것이다.

## 4. 앞으로 어떻게 개선하는가

### 다음 아키텍처: RTL RAG + Spec RAG

```
현재                              다음 단계
┌─────────┐                      ┌─────────┐  ┌─────────┐
│ RTL RAG │  ──── 59% ────→     │ RTL RAG │──│Spec RAG │  ──── 85%+ ────→
│ (How)   │                      │ (How)   │  │ (What)  │
└─────────┘                      └─────────┘  └─────────┘
                                       ↕ 교차 검증
```

| 단계 | 시기 | 내용 | 예상 품질 |
|------|------|------|----------|
| **지금** | 완료 | RTL RAG 단독 운영 | **59%** |
| **Phase 1** | Q3 2026 | Spec RAG 추가 + RTL RAG 고도화 | **85%+** |
| Phase 2 | Q4 2026 | RTL↔Spec 교차 검증, Design Doc 자동 생성 | 엔지니어 검토 후 사용 가능 |
| Phase 3 | Q1 2027 | Safety 분석(FMEDA) 자동화, Signoff 자동화 | Safety 팀 병목 해소 |

## 5. 이걸 해서 뭐가 기대되는가

| 효과 | 설명 |
|------|------|
| **HDD 작성 시간 70% 단축** | 2~4주 → 2~3일 (AI 초안 + 엔지니어 검토) |

### 중기 (Phase 2~3 — Q4 2026~Q1 2027)

| **RTL↔Spec 불일치 자동 감지** | 설계 변경 시 Spec과 구현의 괴리를 조기 발견 |

### 장기 (Phase 4 — Q2 2027)

| 효과 | 설명 |
|------|------|
| **DV 생산성 30%+ 향상** | 검증 Testbench 초기 셋업 자동화 |
| **세대 간 지식 전달** | 기술 지식이 사람이 아닌 시스템에 축적 → 이직 리스크 완화 |
| **사내 기술 Q&A** | "이전 칩에서 이거 어떻게 했지?" 즉시 답변 |

## 6. 무엇이 필요한가

### 인프라: 이미 있다

RTL RAG 구축 과정에서 만든 인프라(OpenSearch, Bedrock, Lambda, MCP Bridge)를 **그대로 재사용**한다. Spec RAG 추가 시 Lambda 파서 코드와 인덱스만 추가하면 된다. 추가 인프라 투자는 최소.

### 사람: 부족하다

| 현재 | 필요 |
|------|------|
| 1인이 RTL RAG 구축 + 튜닝 + 운영을 전부 수행 | RTL RAG 운영과 Spec RAG 구축을 **동시에 할 수 없음** |

**요청: RAG 엔지니어 1명 즉시 충원**

| 시나리오 | Phase 1 완료 시점 | Phase 2 시작 |
|----------|------------------|-------------|
| 현행 1인 | 2026년 말 | 2027 Q1 |
| **2인 체제** | **Q3 2026** | **Q4 2026** |

1명 추가로 **6개월을 앞당길 수 있다.**

### 충원 근거

- 1인이 12일 만에 0→59%를 달성한 생산성이 검증됨
- 구축된 인프라와 방법론이 재사용 가능 → 추가 인력의 생산성이 **즉시 발현**
- RAG Farm은 설계·검증·Safety·FW·품질 **전 팀**에 영향하는 기반 시스템
- 인증 준비 기간 단축은 **프로젝트 일정 리스크 감소**에 직결
