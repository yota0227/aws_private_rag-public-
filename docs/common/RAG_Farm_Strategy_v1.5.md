## 1. 왜 시작했는가

반도체 설계에서 **문서 작성**은 가장 큰 반복 병목이다.

- 블록 HDD 1건 작성에 **2~4주**, 칩 전체로는 **수십 인·월**
- 코드는 바뀌는데 문서가 안 따라감 → 인증(ISO 26262, ASPICE) 직전 병목
- FMEDA, Testbench, Waiver 등 **팀마다 유사한 반복 업무**가 존재

BOS-AI 팀은 이 문제를 **AI 기반 RAG 시스템**으로 해결하고자 RTL 소스 코드 자동 분석 파이프라인을 구축했다.

### 2.4 품질 안정성 — 환각 제거

**RAG 시스템의 존재 이유는 환각 방지다.** 유저가 어떤 프롬프트를 쓰든, 어떤 환경에서 호출하든 **KB에 없는 사실을 지어내지 않는 것**이 핵심.

| 항목 | v5.1 | v6c 이후 |
|------|------|---------|
| Grid 방향 | ❌ 5×4로 뒤집힘 | ✅ 4×5 정확 |
| 비-Tensix 타일 종류 | ❌ ETH/DRAM/PCI 환각 (일반 SoC 스테레오타입) | ✅ 정답지 일치 |
| Flit 폭 | ❌ 116-bit (SECDED 모듈명에서 유추) | ✅ 언급 없음 (KB 부재 시 `[NOT IN KB]`) |

**v6c부터 환각 0건 유지.** 이는 ISO 26262 / ASPICE 같은 인증 맥락에서 결정적 의미를 갖는다 — 자동 생성 산출물이 검증 엔지니어에게 신뢰 가능한 형태로 전달되려면 **"거짓말하지 않음"이 전제 조건**.

## 3. 파이프라인의 성숙 — 깨달음의 과정

13일간 단순히 숫자만 올라간 게 아니라, **RAG 엔지니어링 방법론 자체가 성숙**했다.

### 3.1 개선 레이어가 단계별로 달랐다

| 단계 | 건드린 곳 | 성격 |
|------|----------|------|
| v5→v5.1 | 검색 랭킹 (boost 가중치) | 이미 있는 신호 재발견 |
| v5.1→v6c | KB 데이터 범주 신설 (Package Parser) | 빈 영역 점유 |
| v6c→v7 | KB 데이터 밀도 증가 (Port Classifier) | 기존 영역 두껍게 |
| v7→v8 | **검색 cutoff 확대 (max_results)** | **축적된 KB의 해금** |

### 3.2 v8의 교훈 — 병목은 한 곳에 있지 않다

v7 릴리스 후 "PRTN 포트 14개가 왜 안 잡히지?"라는 문제를 **Port Classifier의 한계로 진단**했다. 실제 원인은 다른 곳이었다 — **검색 결과 상위 20개 cutoff에 가려져 있었을 뿐**, Classifier는 이미 완성돼 있었다. 파라미터 하나(max_results=50)로 해결됐다.

**학습:**
- RAG는 **"공급 측(parser, KB 구축)"**과 **"소비 측(retrieval, 검색)"**의 두 레이어로 구성됨
- 한 레이어의 문제로 진단했을 때, 실제 원인이 다른 레이어에 있을 수 있음
- **두 레이어를 독립적으로 측정**하는 진단 방법론이 필요
- 이 구조적 이해가 생긴 것이 v8까지의 가장 큰 자산

### 3.3 파서 역할 분리 방향

현재는 파서들이 한 덩어리로 묶여 있다. 기여도 측정이 어렵고 확장성에 제약이 있다. 다음 단계에서 **역할별 파서 분리**를 고려 중:

- **Structure** (hierarchy, 인스턴스 트리)
- **Interface** (Port classification, 프로토콜)
- **Type** (localparam, enum, struct) — Package Parser 확장
- **Behavior** (always/assign, pseudo code) — *신규*
- **Connectivity** (iJTAG 체인 등 upstream/downstream 관계) — *신규*

목적: 각 파서의 기여도를 독립 측정 → 다음 투자 포인트 식별

## 4. 현재 한계

**RTL 코드는 "어떻게(How)"만 담고 있다. 설계 의도는 Spec 문서에 있다.**

v8의 68%가 **RTL 단독의 현실적 천장에 근접**하고 있다. 남은 32%를 분석하면:

| 카테고리 | 비중 | 해결 경로 |
|----------|------|----------|
| **Spec-only 정보** | ~50% | Spec RAG 합류 필수 (v9) |
| **RTL에 있지만 깊은 구조** | ~30% | Behavior/Connectivity 파서 추가 (v8.x) |
| **Hybrid territory (서술/추론)** | ~20% | Hybrid 그라운딩 도입 (v8.1) |

구체적 예시:
- **Spec-only:** FPU G-Tile 상세, N1B0 vs Baseline 차이 테이블, AXI 56-bit 주소 맵 의미
- **RTL 깊은 구조:** DOR 알고리즘 (always 블록), EDC U-shape ring (generate 블록)
- **Hybrid:** NoC 표준 동작 원리 + KB 사실 결합

**결론: 80%+ 돌파는 RTL RAG 단독으론 불가. Spec RAG가 필요.**

### 5.2 v8.1 "Hybrid 그라운딩"이란

현재 RAG의 두 모드:
- **Grounded:** KB에 있는 것만 기술, 나머지는 `[NOT IN KB]` → 거짓말 없음, 빈 섹션 많음
- **No Grounding:** 자유 서술 → 문서는 꽉 차지만 과거(v5.1) 환각 발생

**Hybrid 모드 = 제3의 길:**

```markdown
## NoC Routing Algorithm

[GROUNDED from KB]
지원 알고리즘 3종: DOR, Tendril, Dynamic (tt_noc_pkg.sv)

[INFERRED by LLM]
DOR는 일반적으로 X-first 순서로 동작하며, 데드락 방지를 위해
VC 2개 이상을 사용합니다. ※ Spec 확인 필요.
```

검증 엔지니어는 `[GROUNDED]`는 신뢰하고 `[INFERRED]`만 검토하면 됨. **환각 방지와 문서 완결성의 양립**.

## 8. 인력

### 8.1 인프라: 이미 있다

1인이 맨바닥에서 전체 인프라를 직접 구축했다. 신규 RAG 추가 시 **파서 코드 + 인덱스만 추가**하면 즉시 동작한다.

### 8.2 사람: 부족하다

**요청: RAG 엔지니어 1명 즉시 충원**

현재까지 이 업무는 **1인이 기본으로 맡고 다른 멤버가 옵셔널하게** 진행해 온 상태였다. **성과가 입증되고 목표가 구체화된 지금**, 전담 인력 확보가 다음 단계의 전제조건이다.

| 시나리오 | 전체 RAG 구축 | Workflow 자동화 시작 |
|----------|-------------|-------------------|
| 현행 (1인 + 옵션) | 2026년 말 | 2027 Q1 |
| **2인 체제 (즉시 충원)** | **Q2 2026 내** | **Q3 2026** |

| 담당 | 역할 |
|------|------|
| **A (현재)** | RTL RAG 고도화 (v8.1~v9) + Spec RAG 구축 |
| **B (충원)** | Safety/DV/Compliance RAG 파서 개발 + Workflow 아이템 준비 |

### 8.3 충원 근거

- 1인 × 13일로 12% → **68%** — **생산성 검증 완료**
- 인프라 100% 재사용 → **ramp-up 4주 내 본격 기여 가능**
- RAG Farm = 설계·DV·Safety·FW·품질 **전 팀 기반 시스템**
- 인증 준비 단축 = **프로젝트 일정 리스크 감소**
- 기술 지식 **시스템 축적** → 이직 리스크 완화
