## 2. 어떻게 했나 — 4단 전진

### 2.1 v6c (4/28) — 공급 확장

Package Parser 신설. localparam / enum / typedef struct 추출. tile_t enum 8종, Endpoint Table 20행 확보. 50% → 45%*
*(재측정 기준 변경. 상세는 §5 각주.)*

### 2.2 v7 (5/1) — 공급 밀도

Port Classifier 추가 + MCP 응답 200자 → 800자 확장. trinity.sv top-level 포트 전부 노출. 45% → 55%.

### 2.3 v8 (4/29) — 소비 해금

Retrieval cutoff 확대 (max_results 20 → 50). 파서 변경 없이 PRTN(14) / AXI(39) / SFR(17) / EDC IRQ(5) 전부 노출. **55% → 68%.**

병목을 Parser로 오진했다가 Retrieval cutoff 한 줄로 해결된 사건. 여기서 "공급측과 소비측을 분리해 측정해야 한다"는 방법론이 정립되었습니다.

### 2.4 v9 (5/6) — 통합 배선

- **공급측 파서 6종 추가/확장:** Package 함수/태스크, Generate 블록 토폴로지, Always 블록 클럭 도메인, flit 구조 상세, 중첩 struct, 비트폭 표현식 평가
- **소비측 고도화:** 대형 모듈 청킹 (50+ 포트 → Sub_Record), 질의 유형별 동적 Boost (5가지)
- **생성측:** Hybrid Grounding `[GROUNDED]` / `[INFERRED]` 태그 도입 (환각 방지와 문서 완결성 양립)
- **품질 인프라:** 파서별 기여도 독립 측정 프레임워크 (feature flag + CloudWatch)

68% → **측정 예정** (기대 75~78%).

## 3. 그 과정에서 얻은 자산 — 숫자 너머

숫자 외에 2주간 축적된 것:

| 자산 | 설명 |
|------|------|
| 2축 진단 방법론 | "Parser(공급) vs Retrieval(소비)" 분리 측정. v7→v8 병목 오진이 낳은 구조적 통찰 |
| 파서 기여도 독립 측정 | v9에 탑재. 다음 투자 포인트를 숫자로 식별 가능 |
| Hybrid Grounding 태그 | KB 기반과 LLM 추론을 시각적으로 분리. 검증 엔지니어가 "확인 대상"만 보면 됨 |
| 환각 0건 연속 유지 | v6c부터 전 버전. ISO 26262 / ASPICE 인증 맥락에서 결정적 |
| 269개 단위 테스트 | 파서 회귀 방지. 향후 확장 시 안전망 |

## 4. 환각 0이 왜 중요한가

유저가 어떤 프롬프트를 쓰든, 어떤 환경에서 호출하든, KB에 없는 사실을 지어내지 않습니다. 이는 두 가지를 보장합니다.

1. **신뢰성** — 검증 엔지니어가 자동 생성물을 "일단 맞다고 전제"하고 검토 가능
2. **재현성** — 동일 RTL 입력에 대해 백여 명의 엔지니어가 동일한 정보를 얻음. 개인 암묵지를 조직 자산으로 전환

이게 v6c 이후 13일간 0건 유지되고 있다는 점이 2주 성과의 진짜 기반입니다.

## 7. 서비스 배포 계획 — Alpha → Beta → v10

"빨리 유저에게 쓰게 한다"와 "환각으로 신뢰 잃지 않는다"를 동시에 잡기 위해 3단계로 분리 배포합니다. Alpha는 즉시 시작합니다.

### 7.1 Alpha — 즉시 시작 (5월 중)

| 항목 | 내용 |
|------|------|
| 대상 | NPU 분석팀 4인 — 노종호, 임준규, 류승범, 원동언 |
| 범위 | RTL HDD 초안 생성 |
| 가이드 | "초안이며 검토 전제로 사용" |
| 목적 | 실사용 피드백 수집, 환각 0 유지 확인, 사용 패턴 관찰 |

### 7.2 Beta — 5월 말 배포

| 항목 | 내용 |
|------|------|
| 대상 | HQ_DV 팀 — 강성문 + 3~4명 |
| 범위 | HDD 초안 + Q&A |
| 전제조건 | v9.x 가드레일 탑재 완료 |

**v9.x 가드레일(필수):**
1. **빈 검색 결과 Refuse** — hit/score 임계값 이하일 때 "KB에 없음" 강제
2. **서버측 그라운딩 주입** — 유저가 프롬프트를 어떻게 쓰든 시스템이 `[GROUNDED]`/`[INFERRED]` 강제

유저가 백여 명으로 늘기 전에 시스템이 프롬프트 우회를 막지 못하면, 첫 환각 사례 1건이 "AI 못 믿겠다"로 번져 조직 수용성을 영구 훼손할 수 있습니다. Beta 확대 전에 반드시.

### 7.3 v10 — 6월 진행

| 항목 | 내용 |
|------|------|
| 내용 | Spec RAG 합류 (RTL + Spec 교차 검색) |
| 목표 | 80%+ 품질 |
| 이후 일정 | 6월 v10 진행 중 GA·Workflow 아이템 재수립 |

### 7.4 전체 타임라인

```
5월 중 (즉시)     Alpha    NPU 분석팀 4인 (노종호·임준규·류승범·원동언)
                             ↓ 실사용 피드백 수집
5월 내             v9.x     환각 가드레일 탑재 (Refuse + 서버측 그라운딩)
                             ↓
5월 말             Beta     HQ_DV (강성문 + 3~4명)
                             ↓ 가드레일 실전 검증
6월                v10      Spec RAG 합류, 80%+ 달성
                             ↓
(이후 일정은 6월 v10 진행 중 재수립)
```

## 9. 인력

2주간 1인이 Parser·Retrieval·Generation·Quality 4개 레이어를 모두 진전시킨 생산성은 검증되었습니다. 다음 단계(Spec RAG + Workflow 자동화)로 넘어가려면 전담 1인 충원이 전제입니다.

| 시나리오 | RAG Farm 완성 | Workflow 자동화 시작 |
|----------|--------------|--------------------|
| 현행 (1인 + 옵션) | 2026년 말 | 2027 Q1 |
| 2인 체제 (즉시 충원) | Q2 2026 내 | Q3 2026 |

| 담당 | 역할 |
|------|------|
| A (현재) | RTL RAG 고도화 (v9.x) + v10 Spec RAG 구축 + Alpha/Beta 운영 |
| B (충원) | Safety/DV/Compliance RAG 파서 개발 + Workflow 아이템 준비 |

**충원 근거:**
- 1인 × 13일로 50% → 68% — 생산성 검증 완료
- 인프라 100% 재사용 → ramp-up 4주 내 본격 기여 가능
- RAG Farm = 설계·DV·Safety·FW·품질 전 팀 기반 시스템
- 인증 준비 단축 = 프로젝트 일정 리스크 감소
