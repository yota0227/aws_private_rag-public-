## 먼저 알아두면 좋은 것 하나

**RAG에는 두 종류의 데이터가 들어 있습니다.**

| 종류 | 예시 |
|---|---|
| **RTL/설계 코드** | Trinity·N1B0 RTL 파일, 모듈 포트, 신호, 인스턴스, 레지스터 맵 |
| **문서** | 업로드된 HDD·스펙 PDF·설계 문서·주간 보고서 |

모르면 그냥 질문하세요. AI가 어느 쪽에서 찾을지 알아서 결정합니다.

## 자주 쓰는 질문 패턴

### "이 모듈이 뭐야? 포트 알려줘"

```
tt_noc_router 모듈의 포트 목록 알려줘
```
```
BLK_UCIE 모듈의 입출력 신호 정리해줘
```
```
trinity_noc2axi 포트 prefix가 뭘 의미해?
```

> RTL 코드에서 바로 찾아줍니다. 파일 경로나 pipeline ID를 몰라도 됩니다.

---

### "이 신호가 어디로 가?"

```
BLK_UCIE에서 tx_data 신호가 어디까지 전달되는지 추적해줘
```
```
npu_irq 신호 전파 경로 찾아줘
```

---

### "이 모듈 아래 뭐가 있어?"

```
BLK_UCIE 하위 인스턴스 트리 보여줘
```
```
tt_overlay_wrapper 하위 모듈 구조 3단계까지 알려줘
```

---

### "클럭 도메인 크로싱 있어?"

```
BLK_UCIE 모듈에서 클럭 도메인 크로싱 신호 목록 알려줘
```

---

### "업로드된 문서에서 찾아줘"

```
NOC2AXI 라우터 HDD에서 composite tile 설명 찾아줘
```
```
N1B0 성능 모니터 HDD에서 latency 모니터링 방법 알려줘
```
```
UCIE PHY 초기화 시퀀스 문서에서 설명해줘
```

> 올려둔 PDF·마크다운 문서에서 찾습니다. RTL 코드가 아닌 설명 문서를 보고 싶을 때 씁니다.

---

### "뭐가 올라가 있어?"

```
RAG에 어떤 문서가 올라가 있어?
```
```
soc 팀 문서 목록 보여줘
```
```
등록된 팀/카테고리가 뭐야?
```

> 무엇을 물어볼 수 있는지 먼저 파악할 때 씁니다.

---

### "HDD 섹션 만들어줘"

```
ucie/phy/ltssm topic으로 "LTSSM 상태 머신" HDD 섹션 만들어줘
```

> 검증된 claim 기반으로 마크다운 형식 HDD 섹션을 생성합니다.

---

### "Stale HDD 다시 만들어줘"

```
stale HDD 섹션 일괄 재생성해줘
```

> 시간이 걸리는 작업입니다. 바로 결과가 안 나오고 `job://` 형식의 진행 ID를 돌려줍니다. 그걸로 나중에 상태를 확인하세요.

```
job 상태 확인해줘. job_id는 a1b2c3-uuid야
```

---

### "이 답변에 근거가 있어?" (신규)

```
방금 받은 답변이 실제 근거가 있는지 확인해줘
```

> AI가 생성한 답변을 문장별로 검사해서 RTL·문서 근거가 없는 문장을 알려줍니다. HDD 섹션을 검증할 때 유용합니다.

---

### "인덱스가 최신 상태야?" (신규)

```
인덱스 상태 알려줘
```
```
마지막으로 데이터가 업데이트된 게 언제야?
```

> 각 인덱스(버전, 마지막 업데이트 시각, 임베딩 모델)를 목록으로 보여줍니다.

---

### "원문 직접 가져와줘" (신규)

> 답변 말미의 `--- structured ---` 블록에 `resource_uris` 목록이 붙어 있으면, 그 URI로 원문을 직접 조회할 수 있습니다.

```
rtl://npu-rtl/tt_tensix_pkg.sv#L120-L180 원문 가져와줘
```
```
claim://npu-hdd/claim/abc123 내용 보여줘
```

> URI가 `rtl://`, `rag://`, `claim://` 등으로 시작하면 그 형식 그대로 붙여 넣으면 됩니다.

## 잘 안 되면 이렇게 바꿔 보세요

| 증상 | 해결책 |
|---|---|
| "근거를 찾을 수 없습니다"가 나올 때 | RTL 데이터를 물어봤는데 문서 검색으로 들어간 경우입니다. **모듈명·신호명으로 다시 물어보세요.** 예: "noc2axi 관련 모듈 찾아줘" |
| 응답이 너무 일반적일 때 | 프로젝트 이름(Trinity, N1B0)이나 블록명(NoC, AXI, EDC)을 구체적으로 넣어보세요 |
| 최신 데이터가 아닌 것 같을 때 | "인덱스 상태 알려줘"라고 하면 인덱스 버전과 마지막 업데이트 시각을 확인할 수 있습니다 |
| 오류 메시지가 나올 때 | 아래 3가지 에러 코드를 참고하세요 |

**에러 코드 읽는 법:**

| 코드 | 의미 | 대응 |
|---|---|---|
| `invalid_uri` | Resource URI 형식이 잘못됨 | URI를 다시 확인하세요 |
| `not_found` | 요청한 자원·job·claim이 없음 | 이름·ID를 확인하거나 데이터 업로드 여부를 확인하세요 |
| `upstream_error` | 백엔드 Lambda 오류나 타임아웃 | 잠깐 기다렸다 재시도하세요. 반복되면 운영팀에 문의 |

## 이런 건 아직 안 돼요

- 새 RTL 파일이나 문서를 직접 올리는 건 별도 업로드 경로를 써야 합니다
- RTL 코드를 자동으로 수정하거나 패치를 적용하는 것은 지원하지 않습니다
- SoC RTL 데이터는 아직 연동되지 않았습니다 (현재 NPU/Trinity·N1B0 범위)
- **Multi-RAG 도메인 라우팅** — NPU RAG와 SoC RAG를 동시에 검색하거나 corpus를 직접 지정하는 기능은 현재 지원되지 않습니다. SoC RAG 도입 시 MCP 서버에 corpus 라우팅·접근 제어 계층을 추가하는 작업이 선행되어야 합니다 (`.kiro/specs/mcp-corpus-routing-acl/` 에서 설계 예정).

## 프롬프트 팁

- **모듈명은 정확히** 써주면 빠릅니다. 대소문자를 구분하지 않아도 대부분 찾습니다.
- "~를 설명해줘"보다 **"~포트 목록", "~신호 경로"처럼 원하는 것을 구체적으로** 쓰면 더 정확합니다.
- 답변 끝에 출처 정보(`--- structured ---` 블록)가 붙습니다. 일반적으로는 **무시해도 됩니다.** 내용은 그 위쪽을 읽으면 됩니다. 다만 `resource_uris` 항목에 `rtl://` `rag://` `claim://` 등의 URI가 있으면, 그걸 "원문 직접 가져와줘" 패턴에 쓸 수 있습니다.
- **빈 결과가 나와도 에러가 아닙니다.** "해당 데이터가 없다"는 정상 응답입니다.
