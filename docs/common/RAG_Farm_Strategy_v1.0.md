# BOS-AI RAG Farm 전략 보고서 v1.0

**보고일:** 2026-04-28
**작성:** BOS-AI 팀
**대상:** 경영진
**상태:** v1.0 — 성과 검증 완료, 확장 계획 수립

---

## Executive Summary

BOS-AI 팀은 12일간(4/17~4/28) RTL RAG 시스템의 **6회 반복 튜닝**을 통해, 반도체 설계 문서(HDD) 자동 생성 품질을 측정 가능한 수준으로 끌어올렸다. 현재 시스템은 RTL 소스 코드에서 **17개 실제 모듈명과 포트 정보를 정확히 추출**하며, 정답지 대비 **구조 커버리지 59%**에 도달했다.

그러나 정답지의 핵심 항목(패키지 상수, 알고리즘 본문, 토폴로지 다이어그램)을 재현하려면 **RTL RAG 단독으로는 한계**가 명확하다. RTL은 "어떻게(How)"만 알려주고, "무엇을(What)"은 Spec 문서에 있기 때문이다.

다음 단계는 **Spec RAG 구축**이다. 이것이 완성되면 RTL↔Spec 교차 검증, Design Document 자동 생성, TB 코드 생성 등 **7개 후속 활용 아이템**의 기반이 확보된다. 이 확장을 위해 **전담 인력 충원**이 필요하다.

## 1. 현재 도달 수준

### 1.1 RTL RAG 튜닝 이력

| 버전 | 날짜 | 핵심 변경 | 결과 |
|------|------|----------|------|
| v1 | 4/17 | 기본 파싱 + claim + HDD 섹션 | 정답지 대비 ~15% |
| v2 | 4/20 | 프롬프트 토픽별 분할 | 구조 개선, 품질 유사 |

| v4 | 4/27 | analysis_type 패스스루, boost 가중치 도입 | 모듈명 정확성 향상, but 커버리지 하락 |
| **v5.1** | **4/28** | **module_parse 가중치 리밸런싱 (0.3→1.5)** | **커버리지 18%→59%, 14섹션 완전 생성** |

### 1.2 정량 성과 (v5.1 기준)

| 지표 | 수치 |
|------|------|
| KB 확인 모듈 수 | **17개** (SFPU, TDMA, NoC, EDC, Overlay, Clock, SMN) |
| 자체 커버리지 | **59%** (17/29 항목 KB 확인) |
| 정답지 가중 커버리지 | **~13%** (핵심 항목 기준) |
| chip-level HDD 생성 | **14섹션 완전 생성** (494줄) |
| 검색 결과 다양성 | 20건 (HDD 4 + claim 8 + module_parse 8) |
| 인덱싱된 RTL 데이터 | 8,107건 module_parse + 99건 claim + 80건 hierarchy |

### 1.3 무엇이 되고, 무엇이 안 되는가

| 되는 것 (59%) | 안 되는 것 (41%) |
|---------------|-----------------|
| RTL 모듈명 정확 추출 | 패키지 상수 (SizeX, SizeY, tile_t enum) |
| 포트 수/이름 확인 | 알고리즘 본문 (DOR pseudo code, ATT lookup) |
| 모듈 간 인스턴스 관계 | 토폴로지 다이어그램 (EDC ring, clock tree) |
| ECC/CDC 모듈 스펙 | 주소 맵 (AXI 56-bit, APB register offset) |
| 인터럽트 시그널 확인 | 전력 관리 (PRTN chain, ISO_EN) |

**한 줄 진단:** RTL RAG는 **"무엇이 있는가"**는 잘 찾지만, **"그것이 어떻게 동작하는가"**는 Spec 문서 없이 복원할 수 없다.

## 2. 왜 RTL RAG 단독으로는 한계인가

엔지니어가 수작업으로 작성한 정답지(1,291줄)를 분석한 결과:

| 정보 출처 | 정답지 기여도 | RTL RAG 커버 가능 |
|----------|-------------|------------------|
| **Spec 문서** (HDD, AddressMap, I/F Spec) | **~60%** | ❌ |
| **RTL 코드** (모듈 구조, 포트, 인스턴스) | ~25% | ✅ (현재 수준) |
| **엔지니어 도메인 지식** | ~15% | ❌ |

RTL 코드는 설계의 **구현체**일 뿐이다. 설계 **의도**는 Spec 문서에 있다. 정답지의 핵심 섹션(Package Constants, Top-Level Ports, Routing Algorithm, Power Management)은 모두 Spec에서 유래한다.

## 3. 확장 계획: RAG Farm

### 3.1 아키텍처

```
┌──────────────────── RAG Farm ────────────────────────┐
│                                                       │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌────────┐ │
│  │ RTL RAG │  │Spec RAG │  │Safety   │  │DV RAG  │ │
│  │ (운영중)│  │ (다음)  │  │RAG      │  │        │ │
│  └─────────┘  └─────────┘  └─────────┘  └────────┘ │
│                                                       │
│  공유 인프라: OpenSearch AOSS, Bedrock, Lambda, MCP   │
└───────────────────────────────────────────────────────┘
```

### 3.2 로드맵

| Phase | 기간 | 내용 | 기대 효과 |
|-------|------|------|----------|
| **Phase 0** | 완료 | RTL RAG 운영 + 6회 튜닝 | 모듈 구조 자동 추출, HDD 초안 생성 |
| **Phase 1** | Q3 2026 | **Spec RAG 구축** + RTL RAG 고도화 | 정답지 커버리지 13%→**60%+** |
| Phase 2 | Q4 2026 | RTL↔Spec 교차 검증, Design Doc 자동 생성 | 인증 준비 기간 **50% 단축** |
| Phase 3 | Q1 2027 | FMEDA 자동 분석, Waiver 자동화 | Safety signoff **수 주→수 일** |
| Phase 4 | Q2 2027 | TB 자동 생성, Debug KB, 통합 Q&A | DV 생산성 **30%+ 향상** |

### 3.3 Phase 1 상세: Spec RAG + RTL RAG 고도화

| 작업 | 설명 | 효과 |
|------|------|------|
| **Spec RAG 구축** | HDD(Word), AddressMap(Excel), I/F Spec(PDF) 임베딩 | "What" 정보 확보 → 7개 후속 아이템 기반 |
| **Package parser 추가** | localparam, typedef enum, struct 추출 | SizeX/SizeY/tile_t 등 패키지 상수 복원 |
| **KB chunking 확장** | pseudo code, typedef 블록 단위 인덱싱 | 알고리즘 본문, 데이터 구조 정의 복원 |
| **Hybrid 그라운딩** | KB 데이터 + LLM 보강 (태그 구분) | 문서 완전성 + 투명성 동시 확보 |

**Phase 1 완료 시 예상 커버리지:**

| 지표 | 현재 (v5.1) | Phase 1 후 |
|------|------------|-----------|
| 자체 커버리지 | 59% | **85%+** |
| 정답지 가중 커버리지 | ~13% | **60%+** |
| HDD 자동 생성 품질 | 초안 수준 | **엔지니어 검토 후 사용 가능** |

## 5. 인력 충원 필요성

### 5.1 현재 병목

| 병목 | 영향 | 해소 방법 |
|------|------|----------|
| **1인 체제** | RTL RAG 튜닝과 Spec RAG 구축을 동시 진행 불가 | 전담 인력 추가 |
| **Spec 문서 파서 개발** | Word/Excel/PDF 각각 별도 파서 필요 | 파서 개발 인력 |
| **팀별 요구사항 수집** | DV, Safety, FW 팀과 병렬 협의 필요 | PM 또는 추가 엔지니어 |
| **품질 검증 루프** | 정답지 비교, 3-LLM 교차 검증에 시간 소요 | 자동화 + 인력 |

### 5.2 충원 제안

| 역할 | 인원 | 담당 | 시기 |
|------|------|------|------|
| **RAG 엔지니어** | 1명 | Spec RAG 구축, 파서 개발, KB 품질 관리 | 즉시 |
| **MLOps / 평가 자동화** | 1명 | 정답지 기반 자동 채점, regression 방지, 파이프라인 운영 | Phase 1 중 |
| **도메인 엔지니어 (파트타임)** | 0.5명 | DV/Safety 팀 요구사항 번역, 정답지 검증 | Phase 2 시작 시 |

### 5.3 충원 근거

**정량적:**
- 1인이 12일 만에 RTL RAG를 0→59% 커버리지로 끌어올림
- 같은 속도로 Spec RAG를 구축하면 Phase 1 완료까지 약 4~6주
- 하지만 RTL RAG 운영/튜닝과 Spec RAG 구축을 **1인이 동시에 할 수 없음**
- 2인 체제로 전환하면 Phase 1을 **Q3 내 완료** 가능

**정성적:**
- RAG Farm은 **10개 활용 아이템**의 기반 — 설계, DV, Safety, FW, 품질 전 팀에 영향
- 반도체 설계 문서 자동화는 **인증 준비 기간 단축**에 직결 — 프로젝트 일정 리스크 감소
- 현재 구축된 인프라와 방법론은 **재사용 가능** — 추가 인력의 생산성이 즉시 발현
- 세대 간 기술 지식이 **사람이 아닌 시스템에 축적** — 이직 리스크 완화

## 7. 결론

RTL RAG는 **12일 만에 0에서 59% 커버리지**까지 도달했다. 튜닝 루프가 실제로 동작한다는 것이 검증됐고, 정량 평가 방법론도 확립됐다.

그러나 RTL 단독으로는 **정답지의 13%(가중 기준)**만 재현 가능하다. 나머지 87%는 Spec 문서에 있다.

**Spec RAG 구축이 다음 돌파구**이며, 이를 위해 **전담 인력 1명의 즉시 충원**을 요청한다. 2인 체제로 전환하면 Q3 내 Phase 1을 완료하고, Q4부터 Design Document 자동 생성과 RTL↔Spec 교차 검증이 가능해진다.

투자 대비 기대 효과:

| 투자 | 기대 효과 |
|------|----------|
| RAG 엔지니어 1명 (연봉) | 인증 준비 기간 50% 단축 (수십 인·월 절감) |
| | FMEDA 작성 수 주→수 일 (Safety 팀 병목 해소) |
| | DV TB 초기 셋업 50% 단축 |
