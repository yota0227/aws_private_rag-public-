## 1. Overview

### 1.1 칩 목적
Trinity N1B0는 **Tenstorrent**의 AI/ML 추론·학습 겸용 프로세서로, 다수의 **Tensix 컴퓨트 코어**를 2D 메시 NoC로 연결한 타일 기반 아키텍처입니다. 고대역폭 온-칩 SRAM과 유연한 데이터플로우 디스패치를 통해 트랜스포머, CNN 등 다양한 워크로드를 효율적으로 처리합니다.

### 1.2 주요 특징
| 특징 | 설명 |
|------|------|
| **타일 기반 아키텍처** | 4×5 그리드(20 타일 슬롯), 최대 12 Tensix 코어 활성 |
| **Tensix 컴퓨트 코어** | FPU + SFPU + TDMA 기반 데이터 무브먼트 |
| **NoC Fabric** | 2D-mesh, DOR/Tendril/Dynamic 라우팅, 멀티-VC flit 기반 |
| **NIU (Network Interface Unit)** | AXI Bridge, ATT(Address Translation Table), SMN 보안 |
| **EDC (Error Detection & Correction)** | 링 토폴로지 시리얼 버스, 하베스트 바이패스 |
| **Multi-Clock Architecture** | ai_clk, noc_clk, dm_clk, ref_clk 독립 도메인 |
| **Overlay Processor** | Chipyard-기반 RISC-V 오버레이(TTTrinityConfig) |

## 5. Compute Tile (Tensix)

### 5.1 개요
각 Tensix 타일은 독립적인 연산 유닛으로, AI 텐서 연산에 최적화된 데이터패스를 포함합니다.

### 5.2 FPU (Floating-Point Unit)
| 항목 | 내용 |
|------|------|
| 지원 포맷 | FP32, BF16, FP16, TF32 [TBC] |
| 연산 | Fused Multiply-Add, Dot Product |
| 파이프라인 깊이 | [TBC] stages |
| 처리량 | 1 MAC/cycle/FPU [TBC] |

### 5.3 SFPU (Special Function Processing Unit)
| 항목 | 내용 |
|------|------|
| 지원 함수 | exp, log, sigmoid, tanh, rsqrt, reciprocal [TBC] |
| 구현 방식 | LUT + 다항식 근사 |
| 공유 관계 | FPU와 파이프라인 공유, 시분할 동작 |

### 5.4 TDMA (Tensix DMA Engine)
- NoC ↔ L1 Cache 간 비동기 데이터 전송
- 다중 채널 DMA, 스트라이드 어드레싱 지원
- `dm_clk` 도메인에서 독립 동작

### 5.5 L1 Cache (On-Tile SRAM)
| 항목 | 내용 |
|------|------|
| 용량 | [TBC] KB per tile |
| 뱅크 수 | [TBC] |
| 접근 | FPU/SFPU direct read, TDMA write-back |
| ECC | EDC 노드를 통한 에러 검출/정정 |

### 5.6 DEST / SRCB Register Files
| Register File | 용도 | 깊이 | 폭 |
|---------------|------|------|----|
| `DEST` | FPU 출력 누적 결과 저장 | [TBC] entries | [TBC] bits |
| `SRCB` | 피연산자 B 스테이징 | [TBC] entries | [TBC] bits |

## 6. Dispatch Engine

### 6.1 구조
각 Tensix 타일은 **East Dispatch**와 **West Dispatch** 두 개의 독립 디스패치 엔진을 갖습니다.

```
         ┌──────────────────────────┐
  NoC ──►│    Command Queue (CQ)    │
         └──────┬───────────┬───────┘
                │           │
         ┌──────▼──┐   ┌───▼──────┐
         │ East    │   │ West     │
         │Dispatch │   │Dispatch  │
         └──────┬──┘   └───┬──────┘
                │           │
         ┌──────▼──┐   ┌───▼──────┐
         │ FPU/    │   │ TDMA/    │
         │ SFPU    │   │ L1 Ctrl  │
         └─────────┘   └──────────┘
```

### 6.2 명령 분배
| 디스패치 | 담당 유닛 | 설명 |
|----------|-----------|------|
| **East Dispatch** | FPU, SFPU, DEST/SRCB | 연산 명령 디코드 및 실행 |
| **West Dispatch** | TDMA, L1 Cache, NoC I/F | 데이터 무브먼트 명령 관리 |

- 양쪽 디스패치는 **독립 파이프라인**으로 병렬 동작
- 커맨드 큐에서 opcode prefix 기반으로 East/West 라우팅
- 의존성(dependency) 관리를 위한 세마포어 메커니즘 내장

## 7. NoC Fabric

### 7.1 토폴로지
- **2D Mesh** — 4행 × 5열 = 20 노드
- 각 Tensix/NIU/ARC/DRAM 타일에 독립 라우터

### 7.2 라우팅 알고리즘

| Algorithm | 약어 | 설명 |
|-----------|------|------|
| **Dimension-Ordered Routing** | DOR | X-first → Y-next 결정론적 라우팅 (기본) |
| **Tendril Routing** | — | Adaptive load-balancing, 혼잡 회피용 대안 경로 |
| **Dynamic Routing** | — | 런타임 라우팅 테이블 기반 유연 경로 선택 |

### 7.3 Flit 구조

```
┌──────┬──────────┬─────────────┬──────────┐
│ VC ID│ Flit Type│   Payload   │ ECC/CRC  │
│(2-3b)│ (2b)     │ (128b TBC)  │ (Nb)     │
└──────┴──────────┴─────────────┴──────────┘
```

| Flit Type | Value | Description |
|-----------|-------|-------------|
| HEAD | 00 | 라우팅 헤더 + 목적지 좌표 |
| BODY | 01 | 데이터 페이로드 |
| TAIL | 10 | 패킷 종료 마커 |
| HEAD+TAIL | 11 | 단일 flit 패킷 |

### 7.4 Virtual Channel (VC) Buffers
- 라우터당 VC 수: [TBC] (일반적으로 2~4)
- 데드락 방지를 위한 VC 분리: Request VC / Response VC
- 각 VC에 독립적 크레딧 기반 흐름 제어

| 엔트리 수 | [TBC] |
| 페이지 크기 | [TBC] |
| 보호 비트 | R/W/X per entry |

### 8.3 SMN (Secure Mailbox Network) 보안

## 9. Clock Architecture

### 9.1 클럭 도메인 맵

```
              ┌─────────┐
  ext_ref ───►│  PLL_AI  ├───► ai_clk  ──► FPU, SFPU, DEST/SRCB
              └─────────┘
              ┌─────────┐
  ext_ref ───►│ PLL_NOC  ├───► noc_clk ──► NoC Routers, VC Buffers
              └─────────┘
              ┌─────────┐
  ext_ref ───►│ PLL_DM   ├───► dm_clk  ──► TDMA, L1 Cache
              └─────────┘
  ext_ref ────────────────────► ref_clk ──► ARC, EDC, DFX, JTAG
```

### 9.2 도메인별 상세

| Domain | Clock | 주요 소비자 | 비고 |
|--------|-------|-------------|------|
| AI Compute | `ai_clk` | FPU, SFPU, Dispatch, Register Files | 최고 주파수 도메인 |
| NoC | `noc_clk` | Mesh Routers, VC Buffers, NIU Bridge | ai_clk와 비동기 |
| Data Movement | `dm_clk` | TDMA Engine, L1 SRAM | ai_clk 대비 독립 스케일링 |
| Reference | `ref_clk` | ARC, EDC Ring, JTAG, Scan | 저주파, 항상 활성 |

### 9.3 CDC (Clock Domain Crossing)
- RTL에서 확인된 모듈: `TTTrinityConfig_IntSyncAsyncCrossingSink_n1x1`
  - 내부 인스턴스: `TTTrinityConfig_SynchronizerShiftReg_w1_d3` (3-stage synchronizer, width=1)
  - 용도: Overlay RISC-V의 인터럽트 동기화
- 일반 CDC: 2FF 또는 3FF 동기화기, gray-code FIFO

## 11. EDC (Error Detection & Correction)

### 11.1 토폴로지
RTL 파싱 결과 `tt_edc_pkg.sv`에서 다음 인터페이스가 확인됨:

```
tt_edc_pkg interfaces (modports):
  ├── ingress   — 업스트림 노드로부터 요청 수신
  ├── egress    — 다운스트림 노드로 요청 전달
  ├── edc_node  — 개별 타일의 EDC 로직 연결
  └── sram      — SRAM 인스턴스 ECC 인터페이스
```

### 11.2 링 토폴로지

```
┌────────┐    ┌────────┐    ┌────────┐         ┌────────┐
│ Tile 0 │───►│ Tile 1 │───►│ Tile 2 │───► … ──►│ Tile N │──┐
│(edc_   │◄───│(edc_   │◄───│(edc_   │◄── … ◄──│(edc_   │  │
│ node)  │    │ node)  │    │ node)  │         │ node)  │  │
└────────┘    └────────┘    └────────┘         └────────┘  │
     ▲                                                      │
     └──────────────────────────────────────────────────────┘
                         Ring Return Path
```

### 11.3 시리얼 버스 프로토콜

| Signal | Direction (ingress) | Description |
|--------|---------------------|-------------|
| `req_tgl` | input | 요청 토글 — 새 트랜잭션 시작 표시 |
| `ack_tgl` | output | 응답 토글 — 트랜잭션 완료 확인 |
| `cor_err` | input | Correctable error 플래그 수신 |
| `err_inj_vec` | output | Error injection 벡터 (DFX 테스트용) |

- **토글 기반 핸드셰이크**: `req_tgl` 전환 → 노드 처리 → `ack_tgl` 전환
- 대역폭 최소화를 위해 1-bit 시리얼 전송
- `ref_clk` 도메인에서 동작

### 11.4 하베스트 바이패스
- 결함 타일 하베스트 시 EDC 링이 단절되지 않도록 바이패스 MUX 삽입
- 하베스트된 타일의 `ingress` → `egress` 직접 연결(edc_node 스킵)
- ARC 펌웨어가 하베스트 맵을 기반으로 바이패스 레지스터 설정
