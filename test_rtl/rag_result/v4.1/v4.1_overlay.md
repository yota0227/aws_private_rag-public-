## 1. Overview

### 1.1 What is the Overlay?

The **Overlay** is Trinity N1B0's RISC-V CPU subsystem вҖ” the programmable control plane that orchestrates all data movement and compute operations across the chip. It is the "brain" that:

- **Dispatches work** to Tensix compute tiles via the Dispatch Engine.
- **Manages DMA transfers** between host memory, DRAM, and tile-local L1 caches.
- **Runs firmware/kernels** on a cluster of RISC-V cores for task scheduling, synchronization, and device management.
- **Provides debug/control** interfaces via APB register slaves and the SMN security network.

The Overlay is **not** a simple CPU вҖ” it is a complete SoC-in-a-tile that integrates an 8-core RISC-V cluster, L1 cache hierarchy, DMA engine, custom accelerators (ROCC), and NoC bridge logic into a single tile position within the Trinity 4Г—5 mesh.

### 1.2 Key Design Philosophy

The Overlay acts as **glue logic** binding four major subsystems:

| Subsystem | Role |
|---|---|
| **CPU Cluster** | Programmable control вҖ” runs firmware, schedules tasks |
| **NoC Interface** | Connects Overlay to the chip-wide Network-on-Chip fabric |
| **L1 Cache / SRAM** | Fast local storage for firmware code and data |
| **DMA Engine** | Bulk data movement without CPU intervention |

### 1.3 RTL Evidence

The RTL search confirms the Overlay contains the module `BDAed_trinity_noc2axi_router_nw_opt_trinity_noc2axi_router_nw_opt_tt_mem_wrap_32x1024_2p_nomask_noc_routing_translation_selftest`, which reveals:

- **`noc2axi_router`** вҖ” a NoC-to-AXI bridge/router, confirming the Overlay bridges NoC flits to AXI transactions for the CPU cluster.
- **`nw_opt`** вҖ” northwest-optimized routing variant, indicating the Overlay's position and routing bias in the mesh.
- **`tt_mem_wrap_32x1024_2p_nomask`** вҖ” a 32-bit Г— 1024-depth dual-port SRAM wrapper (no write mask), used for routing translation tables or small caches.
- **`noc_routing_translation_selftest`** вҖ” built-in self-test for the NoC routing/address translation logic.

This module name encodes the Overlay's core architectural function: **bridging NoC traffic to AXI-based CPU/memory subsystems with built-in address translation and DFT.**

### 2.2 Position Rationale

- **Northwest corner** placement minimizes hop count to the DRAM controllers (right edge) and ARC controller (southwest).
- The `nw_opt` router variant enables **dimension-ordered routing (DOR)** with preferential northwest paths for return traffic.
- Overlay connects to **NoC Plane 0 and Plane 1** for full-duplex communication with all tiles.

| Bank width | 32 bits (matches `tt_mem_wrap_32x1024_2p_nomask` вҖ” 32-bit data port) |
| Bank depth | 1024 entries per macro (4 KB per bank, with multiple macros per bank) |
| Port type | **Dual-port** (2P) вҖ” 1 read + 1 write simultaneously |
| Write mask | **None** (`nomask`) вҖ” full-word writes only |
| ECC | SECDED per bank (encode on write, check+correct on read) |
| SRAM macro | `tt_mem_wrap_32x1024_2p_nomask` |
| EDC integration | `cor_err` вҶ’ EDC ring for correctable error reporting |

#### 6.2.3 SRAM Organization

```
L1 SRAM Scratchpad (1 MB total)
в”ңв”Җв”Җ Bank 0:  tt_mem_wrap_32x1024_2p_nomask Г— M macros  (32 KB)
в”ңв”Җв”Җ Bank 1:  tt_mem_wrap_32x1024_2p_nomask Г— M macros  (32 KB)
в”ңв”Җв”Җ ...
в””в”Җв”Җ Bank 31: tt_mem_wrap_32x1024_2p_nomask Г— M macros  (32 KB)

Access: Address[14:10] selects bank, Address[9:0] selects entry within bank
```

### 6.5 LLK (Low-Latency Kernel)

| Property | Value |
|---|---|
| Full name | **Low-Latency Kernel Engine** |
| Purpose | Hardware-assisted fast kernel dispatch to Tensix tiles |
| Key feature | Bypasses full firmware scheduling for latency-critical operations |
| Mechanism | Pre-programmed kernel descriptors in LLK registers; hardware triggers dispatch on event |

#### 6.5.1 Sub-blocks

| Module | Function |
|---|---|
| `tt_llk_dispatch` | Monitors trigger conditions (semaphore, timer, interrupt), fires pre-loaded kernel descriptors to Dispatch Engine |
| `tt_llk_sync` | Hardware barrier and semaphore вҖ” allows tiles to signal completion without CPU intervention |
| `tt_llk_perf_cnt` | Performance counters вҖ” kernel launch latency, barrier wait cycles, dispatch throughput |

#### 6.5.2 LLK Operation Flow

```
1. CPU pre-loads kernel descriptor into LLK registers (APB write)
2. CPU arms the LLK trigger (e.g., "dispatch when semaphore X reaches N")
3. LLK hardware monitors trigger condition
4. On trigger: LLK вҶ’ Dispatch Engine вҶ’ Tensix tile (sub-microsecond latency)
5. Tensix tile signals completion via LLK sync barrier
6. LLK optionally chains to next kernel descriptor
```

### 6.6 SMN (System Maintenance Network)

| Property | Value |
|---|---|
| Full name | **System Maintenance Network** |
| Purpose | Security and access control for register/memory accesses |
| Mechanism | Address-range firewall вҖ” blocks unauthorized transactions |

#### 6.6.1 Architecture

| Module | Function |
|---|---|
| `tt_smn_access_ctrl` | Compares transaction address against programmed ranges; issues ALLOW/DENY |
| `tt_smn_firewall` | Stores firewall rules (base, limit, permissions per master ID) |
| `tt_smn_apb_bridge` | Provides APB programming interface for rule configuration |

#### 6.6.2 Security Model

- Each AXI/NoC transaction carries a **master ID** identifying the source (CPU core N, DMA, external host).
- SMN checks `(master_id, address)` against firewall rules before the transaction reaches the target.
- Unauthorized accesses are **silently dropped** with an error response on the AXI bus; a security interrupt is raised to PLIC.

| Property | Value |
|---|---|
| Full name | **Frequency / Droop Sensor** |
| Purpose | Monitors on-chip voltage droop and frequency stability |
| Sensor type | **Ring oscillator** вҖ” frequency varies with voltage/temperature |

#### 6.7.1 Operation

| Module | Function |
|---|---|
| `tt_fds_ring_osc` | Configurable ring oscillator вҖ” generates a frequency proportional to local VDD |

4. Firmware can throttle workload or adjust DVFS settings in response.

