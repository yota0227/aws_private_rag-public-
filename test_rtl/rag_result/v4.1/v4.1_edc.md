## 1. Overview

### 1.1 What is EDC1?

**EDC1** (Error Detection & Correction, version 1) is Trinity N1B0's dedicated serial diagnostic and error-management subsystem. It forms a chip-wide ring network that connects every tile — Tensix compute, DRAM controllers, ARC controller, and NoC routers — for the purposes of:

- **ECC error reporting** — correctable (`cor_err`) and uncorrectable error injection/collection.
- **Register access** — lightweight out-of-band register read/write to every tile without NoC traffic.
- **Harvest/fuse management** — runtime discovery and bypass of disabled (harvested) tiles.

### 1.2 Key Characteristics

| Property | Value |
|---|---|
| Bus type | Serial, toggle-handshake (clock-less transfer) |
| Data width | **16-bit data + 1-bit parity** (`data[15:0]`, `data_p`) |
| Handshake | `req_tgl` / `ack_tgl` — level-toggling, no dedicated clock required on the link |
| Topology | Daisy-chain **ring** (U-shape: Segment A ↓ → U-turn → Segment B ↑) |
| Addressing | Hierarchical: `node_id_part` / `node_id_subp` / `node_id_inst` |
| Error injection | Per-SRAM `err_inj_vec` for manufacturing/DFT test |
| Error collection | Per-SRAM `cor_err` flags aggregated to ring head |
| Package definition | `tt_edc_pkg.sv` — defines interfaces `ingress`, `egress`, `edc_node`, `sram` as SystemVerilog modports |

### 1.3 EDC1 vs. Legacy EDC

EDC1 replaces the earlier register-mapped polling approach with a fully asynchronous toggle-based serial bus, eliminating the need for a dedicated EDC clock domain on every link segment.

### 2.2 Data Flow Summary

1. **Ring Head** initiates a transaction by toggling `req_tgl` with a 16-bit command/address fragment.
2. Each **Node** receives via `ingress`, processes or forwards via `edc_node` logic, and outputs via `egress`.
3. **SRAM modport** connects the node to local memory instances for error injection (`err_inj_vec`) and correctable error reporting (`cor_err`).
4. Response travels back around the ring to the Ring Head.

## 3. Serial Bus Interface

### 3.1 Signal Table

The EDC1 serial bus is defined in `tt_edc_pkg.sv` via SystemVerilog interface modports. The signals are:

| Signal | Direction (ingress) | Direction (egress) | Width | Description |
|---|---|---|---|---|
| `req_tgl` | **input** | **output** | 1 | Request toggle — each transition (0→1 or 1→0) indicates a new valid data beat |
| `ack_tgl` | **output** | **input** | 1 | Acknowledge toggle — receiver toggles after consuming the data beat |
| `data` | input | output | 16 | Payload data |
| `data_p` | input | output | 1 | Odd parity over `data[15:0]` |
| `cor_err` | **input** (from sram) | **output** (to sram) | varies | Correctable error flag(s) from local SRAM instance(s) |
| `err_inj_vec` | **output** (to sram) | **input** (from sram) | varies | Error injection vector — used by DFT to flip bits in SRAM read data |
| `async_init` | input | — | 1 | Asynchronous initialization pulse — resets toggle state on the link without requiring a clock |

### 3.2 Toggle-Handshake Protocol

```
Sender (egress)                    Receiver (ingress)
─────────────────                  ─────────────────
  data ← payload
  req_tgl ← ~req_tgl   ───────►   detect req_tgl transition
                                    latch data
                                    ack_tgl ← ~ack_tgl  ───────►  sender sees ack
  (ready for next beat)
```

**Key properties:**
- No shared clock required between adjacent nodes — the toggle encodes "new data available."
- Throughput is limited by round-trip toggle latency (~2–4 ns per hop at typical wire delay).
- Parity (`data_p`) provides single-bit error detection on every beat.

### 3.3 Async Init

`async_init` is a one-shot pulse driven from the reset controller that forces both `req_tgl` and `ack_tgl` to a known state (typically `0`) at power-on or warm reset, ensuring no spurious transactions propagate through the ring.

## 4. Packet Format

### 4.1 Fragment Structure

EDC1 packets are transmitted as a sequence of **16-bit fragments** on the serial bus. A complete packet consists of:

| Fragment # | Field | Bits | Description |
|---|---|---|---|
| 0 | **Header** | `[15:12]` opcode, `[11:0]` node_id | Identifies the target node and operation type |
| 1 | **Address** | `[15:0]` reg_addr | Register address within the target node |
| 2 | **Data Lo** | `[15:0]` wdata[15:0] | Lower 16 bits of write data (write ops) |
| 3 | **Data Hi** | `[15:0]` wdata[31:16] | Upper 16 bits of write data (write ops) |
| … | **Extension** | varies | Additional fragments for burst or vector operations |

### 4.2 Opcode Encoding (Header[15:12])

| Opcode | Mnemonic | Description |
|---|---|---|
| `4'h0` | `NOP` | No operation — used for ring liveness check |
| `4'h1` | `RD` | Register read request |
| `4'h2` | `WR` | Register write request |
| `4'h3` | `RD_RSP` | Read response (data returning to ring head) |
| `4'h4` | `ERR_RPT` | Error report — node pushes `cor_err` status upstream |
| `4'h5` | `ERR_INJ` | Error injection — ring head programs `err_inj_vec` |
| `4'hF` | `BCAST` | Broadcast — all nodes process (used for global init) |

### 4.3 MAX_FRGS

The maximum number of fragments per packet is implementation-parameterized:

| Parameter | Typical Value | Notes |
|---|---|---|
| `MAX_FRGS` | **4** (single register R/W) | Header + Addr + DataLo + DataHi |
| `MAX_FRGS` (burst) | **8** | For SRAM dump / vector error inject |

Fragment count is implicitly determined by the opcode; there is no explicit length field. Each node's ingress logic uses a small FSM to count expected fragments.

## 5. Node ID Structure

### 5.1 Encoding

The 12-bit `node_id` in the packet header is hierarchically structured:

| Field | Bits | Width | Description |
|---|---|---|---|
| `node_id_part` | `[11:8]` | 4 | **Partition** — major functional block (Tensix array, DRAM, ARC, etc.) |
| `node_id_subp` | `[7:4]` | 4 | **Sub-partition** — column or sub-block within the partition |
| `node_id_inst` | `[3:0]` | 4 | **Instance** — specific node within the sub-partition |

### 5.2 Decoding Table (Trinity N1B0)

| `node_id_part` | Partition | `node_id_subp` range | `node_id_inst` range | Total Nodes |
|---|---|---|---|---|
| `4'h0` | Tensix Column 0 | `0–3` (row) | `0` | 4 |
| `4'h1` | Tensix Column 1 | `0–3` (row) | `0` | 4 |
| `4'h2` | Tensix Column 2 | `0–3` (row) | `0` | 4 |
| `4'h3` | DRAM Controller | `0` | `0–1` (ch) | 2 |
| `4'h4` | ARC Controller | `0` | `0` | 1 |
| `4'h5` | NoC Router Array | `0–4` (col) | `0–3` (row) | 20 |
| `4'h6` | Ethernet/PCI (if present) | `0` | `0–1` | 2 |
| `4'hF` | **Broadcast** | `—` | `—` | All |

### 5.3 Address Match Logic

Each node compares incoming `node_id` against its hardcoded parameters:

```systemverilog
wire id_match = (frag0_part == MY_PART) &&
                (frag0_subp == MY_SUBP) &&
                (frag0_inst == MY_INST);

wire is_bcast = (frag0_part == 4'hF);

wire consume  = id_match | is_bcast;
```

If `consume` is low, the node forwards the packet to `egress` without modification.

## 7. Ring Topology

### 7.1 U-Shape Ring

The EDC1 ring follows a **U-shape** path through the 4×5 grid:

```
         Col 0       Col 1       Col 2       Col 3       Col 4
        ┌────────┬────────┬────────┬────────┬────────┐
Row 3   │ Node   │ Node   │ Node   │ Node   │ Node   │  ◄── Segment A
        │ [3,0]  │ [3,1]  │ [3,2]  │ [3,3]  │ [3,4]  │      (Top → Down)
        │   │    │   │    │   │    │   │    │   │    │
        │   ▼    │   ▼    │   ▼    │   ▼    │   ▼    │
Row 2   │ [2,0]  │ [2,1]  │ [2,2]  │ [2,3]  │ [2,4]  │
        │   │    │   │    │   │    │   │    │   │    │
        │   ▼    │   ▼    │   ▼    │   ▼    │   ▼    │
Row 1   │ [1,0]  │ [1,1]  │ [1,2]  │ [1,3]  │ [1,4]  │
        │   │    │   │    │   │    │   │    │   │    │
        │   ▼    │   ▼    │   ▼    │   ▼    │   ▼    │
Row 0   │ [0,0]  │ [0,1]  │ [0,2]  │ [0,3]  │ [0,4]  │  ◄── U-turn row
        └───┼────┴───┼────┴───┼────┴───┼────┴───┼────┘
            └────────┴────────┴────────┴────────┘
                         U-TURN BUS
            ┌────────┬────────┬────────┬────────┐
        ┌───┼────┬───┼────┬───┼────┬───┼────┬───┼────┐
Row 0   │ [0,0]' │ [0,1]' │ [0,2]' │ [0,3]' │ [0,4]' │  ◄── Segment B
        │   ▲    │   ▲    │   ▲    │   ▲    │   ▲    │      (Bottom → Up)
        │   │    │   │    │   │    │   │    │   │    │
Row 1   │ [1,0]' │ [1,1]' │ [1,2]' │ [1,3]' │ [1,4]' │
        │   ▲    │   ▲    │   ▲    │   ▲    │   ▲    │
        │   │    │   │    │   │    │   │    │   │    │
Row 2   │ [2,0]' │ [2,1]' │ [2,2]' │ [2,3]' │ [2,4]' │
        │   ▲    │   ▲    │   ▲    │   ▲    │   ▲    │
        │   │    │   │    │   │    │   │    │   │    │
Row 3   │ [3,0]' │ [3,1]' │ [3,2]' │ [3,3]' │ [3,4]' │
        └────────┴────────┴────────┴────────┴────────┘
                              │
                    Back to Ring Head
```

**Segment A** — packets flow **downward** (row 3 → row 0) across all columns.  
**U-turn** — at row 0, the bus reverses direction.  
**Segment B** — packets flow **upward** (row 0 → row 3) and return to the Ring Head.

### 7.2 Ring Ordering

Within each segment, columns are visited in order `Col 0 → Col 1 → … → Col 4`, then the next row. This means the full traversal order for Segment A is:

```
[3,0] → [3,1] → [3,2] → [3,3] → [3,4] → [2,0] → [2,1] → … → [0,4] → U-turn
```

Total ring length: **2 × (4 rows × 5 cols) = 40 node hops** (plus ring head).

## 8. Harvest Bypass

### 8.1 Purpose

When a tile is disabled (harvested) due to manufacturing defects, its EDC node must be bypassed so the ring remains intact. EDC1 uses a **MUX/DEMUX bypass** mechanism controlled by fuse-programmed select signals.

### 8.2 Bypass Architecture

```
                  Normal Path                    Bypass Path
                  ───────────                    ───────────
  ingress ──►┌────────────┐                 ┌──────────────────┐
             │  DEMUX     │──► edc_node ──►│      MUX          │──► egress
             │            │                 │                    │
             │  sel=0     │                 │  sel=0 (normal)    │
             └────────────┘                 └──────────────────┘
                  │                                   ▲
                  │         sel=1 (bypass)             │
                  └───────────────────────────────────┘
```

### 8.3 Control Signal: `edc_mux_demux_sel`

| `edc_mux_demux_sel` | Behavior |
|---|---|
| `0` (default) | **Normal** — ingress goes through `tt_edc1_node`, then to egress |
| `1` (harvested) | **Bypass** — ingress directly connects to egress, node is isolated |

### 8.4 Programming

- `edc_mux_demux_sel` is set during boot by the ARC controller based on **fuse/eFuse harvest map**.
- The signal is latched and **cannot** be changed at runtime (static configuration).
- When bypassed, the node's `req_tgl`/`ack_tgl` are held in reset state to avoid spurious toggles.

### 8.5 Relevant Modules

| Module | Role |
|---|---|
| `tt_edc1_bypass_mux` | Selects between node egress and direct bypass for output |
| `tt_edc1_bypass_demux` | Selects between node ingress and direct bypass for input |

## 9. BIU (Bus Interface Unit)

### 9.1 Register Access Path

The EDC1 BIU provides the software-visible register interface between the ARC controller and the EDC ring:

```
  ARC CPU  ──►  APB/AHB Bus  ──►  tt_edc1_biu  ──►  tt_edc1_ring_head  ──►  Ring
                                      ▲
                                      │
                                   Register Map
```

### 9.2 BIU Register Map

| Offset | Register | Width | Access | Description |
|---|---|---|---|---|
| `0x00` | `EDC_CMD` | 32 | W | Command register — opcode + target node_id |
| `0x04` | `EDC_ADDR` | 32 | W | Target register address within the node |
| `0x08` | `EDC_WDATA` | 32 | W | Write data |
| `0x0C` | `EDC_RDATA` | 32 | R | Read data — valid after read completion |
| `0x10` | `EDC_STATUS` | 32 | R | Status — `[0]` busy, `[1]` error, `[2]` timeout |
| `0x14` | `EDC_ERR_LOG` | 32 | R/W1C | Error log — parity error location, node ID |
| `0x18` | `EDC_HARVEST` | 32 | R | Harvest bitmap — which nodes are bypassed |
| `0x1C` | `EDC_INJ_CTRL` | 32 | W | Error injection control — target SRAM, vector |

### 9.3 Transaction Flow (Software Perspective)

1. SW writes `EDC_CMD` with opcode=`RD`, node_id=target.
2. SW writes `EDC_ADDR` with the target register offset.
3. BIU serializes into fragments and sends to ring head TX.
4. Ring head transmits fragments around the ring.
5. Target node responds; response returns via ring.
6. Ring head RX deserializes and writes `EDC_RDATA`.
7. `EDC_STATUS.busy` clears → SW reads `EDC_RDATA`.

## 10. CDC / Synchronization

### 10.1 Clock Domain Context

The EDC1 serial bus is **inherently asynchronous** (toggle-handshake), so inter-node links do not require CDC. However, CDC is needed at the **boundaries** where the EDC interfaces with clocked domains:

| Boundary | Source Domain | Target Domain | Mechanism |
|---|---|---|---|
| Ring Head ↔ BIU | `arc_clk` (ARC bus clock) | Toggle domain | `tt_edc1_cdc_sync` — 2FF synchronizer on `ack_tgl` returning to ring head |
| Node ↔ SRAM | Toggle domain | `ai_clk` (tile clock) | `tt_edc1_cdc_sync` — 2FF synchronizer on `cor_err` and `err_inj_vec` |
| Node ↔ Local Regs | Toggle domain | `ai_clk` | Pulse synchronizer for register write strobe |

### 10.2 Synchronizer Module

```
tt_edc1_cdc_sync
├── Parameter: STAGES (default = 3, matching TTTrinityConfig_SynchronizerShiftReg_w1_d3)
├── Input:  async_in   (from toggle domain)
├── Input:  clk        (destination clock)
├── Input:  rst_n      (destination reset)
├── Output: sync_out   (synchronized to clk)
└── Implementation: Shift register chain (d3 = 3 flip-flops)
```

**Reference:** The Chipyard-generated synchronizer `TTTrinityConfig_SynchronizerShiftReg_w1_d3` (found in `TTTrinityConfig_IntSyncAsyncCrossingSink_n1x1.sv`) uses the same 3-stage shift register pattern for 1-bit async crossing, confirming the CDC depth.

### 10.3 `async_init` Reset Synchronization

At chip reset, `async_init` is pulsed to all nodes simultaneously. Each node's toggle state machine resets to `req_tgl=0, ack_tgl=0`. Since this is an asynchronous pulse, it must be wide enough to be captured by the slowest node (~10 ns minimum pulse width).

## Appendix A — Interface Quick Reference

```systemverilog
// From tt_edc_pkg.sv
interface tt_edc_if;
    logic        req_tgl;
    logic        ack_tgl;
    logic [15:0] data;
    logic        data_p;
    logic        cor_err;
    logic        err_inj_vec;

    modport ingress  (input  req_tgl, output ack_tgl, input  cor_err, output err_inj_vec);
    modport egress   (output req_tgl, input  ack_tgl, output cor_err, input  err_inj_vec);
    modport edc_node (input  req_tgl, output ack_tgl, output req_tgl, input  ack_tgl,
                      input  cor_err, output err_inj_vec, output cor_err, input  err_inj_vec);
    modport sram     (input  err_inj_vec, output cor_err);
endinterface
```

## Appendix B — Glossary

| Term | Definition |
|---|---|
| **EDC** | Error Detection and Correction |
| **Toggle-handshake** | Asynchronous signaling where each level transition (not level itself) indicates a new event |
| **Harvest** | Disabling defective tiles via fuse programming |
| **Ring Head** | The master node that initiates all EDC transactions |
| **BIU** | Bus Interface Unit — bridges ARC register bus to EDC ring |
| **cor_err** | Correctable error flag from SRAM ECC logic |
| **err_inj_vec** | Error injection vector — forces bit flips for DFT testing |
| **CDC** | Clock Domain Crossing |
| **Flit** | (NoC term, not used in EDC; EDC uses "fragment") |
| **Fragment** | A single 16-bit + parity beat on the EDC serial bus |
