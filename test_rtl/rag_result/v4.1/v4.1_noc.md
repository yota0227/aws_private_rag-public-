## 1. Overview

### 1.1 What is the Trinity NoC?

The **Network-on-Chip (NoC)** is Trinity N1B0's primary on-chip communication fabric. It connects every tile in the 4Г—5 mesh вҖ” Tensix compute tiles, Overlay (RISC-V), DRAM controllers, ARC controller, and PCIe/Ethernet endpoints вҖ” through a packet-switched, credit-based 2D mesh network.

### 1.2 Key Architectural Characteristics

| Property | Value |
|---|---|
| Topology | **2D Mesh** вҖ” 4 rows Г— 5 columns = 20 router nodes |
| Planes | **2** вҖ” Plane 0 (request/write) + Plane 1 (response/read) |
| Flit width | **64 bytes** (512 bits) per flit |
| Routing | **3 algorithms** вҖ” DIM_ORDER (XY), TENDRIL, DYNAMIC |
| Virtual Channels | **4 VCs** per port direction |
| Arbitration | **Tree-based** (`noc_arbiter_tree`) вҖ” round-robin at each level |
| Flow control | Credit-based with VC-level credits |
| Address width | **56 bits** via AXI address gasket |
| Security | `tt_noc_sec_fence_edc_wrapper` вҖ” SMN group-based access control |
| Inter-column links | **Cardinal repeaters** (`tt_noc_repeaters_cardinal`) between rows 3вҖ“4 |
| Bridge to CPU | `noc2axi_router_nw_opt` вҖ” NoC-to-AXI with routing translation + selftest |

## 2. Routing Algorithms

### 2.1 Three Routing Modes

Trinity NoC supports three routing algorithms, selectable per-packet via the flit header. This enables a trade-off between determinism, latency, and load balancing.

### 2.2 Comparison Table

| Property | **DIM_ORDER (XY)** | **TENDRIL** | **DYNAMIC** |
|---|---|---|---|
| Algorithm | Dimension-ordered: X first, then Y | Source-routed with pre-computed waypoints | Adaptive: selects least-congested output port |
| Determinism | **Fully deterministic** вҖ” same srcвҶ’dst always same path | **Deterministic per route** вҖ” path fixed at source | **Non-deterministic** вҖ” path varies with congestion |
| Deadlock freedom | Guaranteed (XY ordering prevents cyclic dependencies) | Guaranteed (waypoints form acyclic graph) | Requires VC partitioning to avoid deadlock |
| Latency (typical) | Low (minimal routing decision) | Medium (waypoint lookup) | Variable (congestion-dependent) |
| Throughput | Moderate вҖ” hot spots possible on shared X/Y channels | High вҖ” load spreads via pre-planned paths | Highest вҖ” adapts to real-time traffic |
| Use case | Register access, control traffic | Bulk data transfers (DMA, tensor moves) | Mixed workloads, high-utilization scenarios |
| Header encoding | `routing_mode = 2'b00` | `routing_mode = 2'b01` + waypoint list | `routing_mode = 2'b10` |
| Implementation | Simple comparator (current_x vs dest_x, then current_y vs dest_y) | Waypoint table in each router (or source-appended) | Congestion counters per output port + min-selector |

### 2.3 Routing Decision Logic (Simplified)

```
// DIM_ORDER (XY)
if (current_x != dest_x)
    route to East or West (toward dest_x)
else if (current_y != dest_y)
    route to North or South (toward dest_y)
else
    deliver to local endpoint

// TENDRIL
next_hop = waypoint_table[waypoint_index]
waypoint_index++
if (waypoint_index == num_waypoints)
    deliver to local endpoint

// DYNAMIC
congestion[EAST]  = credit_count[EAST]    // lower credits = more congested
congestion[WEST]  = credit_count[WEST]
congestion[NORTH] = credit_count[NORTH]
congestion[SOUTH] = credit_count[SOUTH]
// Select productive direction with lowest congestion
route to min_congestion(productive_directions)
```

### 2.4 Deadlock Avoidance in DYNAMIC Mode

Dynamic routing uses **VC partitioning** to prevent deadlocks:

| VC | Allowed Turns | Purpose |
|---|---|---|
| VC 0 | X-only (East/West) | Horizontal phase |
| VC 1 | Y-only (North/South) | Vertical phase |
| VC 2 | Any productive direction | Adaptive (deadlock escape via VC escalation) |
| VC 3 | Any direction | Response/completion channel (priority) |

A packet can escalate from VC 2 вҶ’ VC 3 if VC 2 is congested, but cannot descend, ensuring acyclicity.

## 3. Flit Structure

### 3.1 `noc_header_address_t` Fields

Each NoC flit carries a structured header. The **header flit** (first flit of a packet) contains the routing and addressing information:

| Field | Bit Range | Width | Description |
|---|---|---|---|
| `x_dest` | `[511:508]` | 4 | Destination X coordinate (column 0вҖ“4) |
| `y_dest` | `[507:504]` | 4 | Destination Y coordinate (row 0вҖ“3) |
| `endpoint_id` | `[503:498]` | 6 | Target endpoint within the tile (CPU port, L1, DMA, etc.) |
| `flit_type` | `[497:496]` | 2 | Flit type: `00`=Header, `01`=Body, `10`=Tail, `11`=Header+Tail (single flit) |
| `routing_mode` | `[495:494]` | 2 | `00`=DIM_ORDER, `01`=TENDRIL, `10`=DYNAMIC, `11`=Reserved |
| `vc_id` | `[493:492]` | 2 | Virtual Channel ID (0вҖ“3) |
| `packet_id` | `[491:484]` | 8 | Packet identifier (for reordering / matching) |
| `src_x` | `[483:480]` | 4 | Source X coordinate |
| `src_y` | `[479:476]` | 4 | Source Y coordinate |
| `src_endpoint` | `[475:470]` | 6 | Source endpoint ID |
| `opcode` | `[469:464]` | 6 | Transaction opcode (Read, Write, Atomic, Response, etc.) |
| `length` | `[463:456]` | 8 | Packet length in flits (1вҖ“255, 0=256) |
| `address` | `[455:400]` | 56 | AXI-mapped address (see Section 4) |
| `reserved` | `[399:384]` | 16 | Reserved for future use |
| `payload` | `[383:0]` | 384 | Data payload (header flit carries first 48 bytes of data) |

### 3.2 Flit Types

| `flit_type` | Mnemonic | Description |
|---|---|---|
| `2'b00` | **HEAD** | First flit вҖ” contains full header + partial payload |
| `2'b01` | **BODY** | Middle flit вҖ” 64 bytes of pure data payload |
| `2'b10` | **TAIL** | Last flit вҖ” final payload + end-of-packet marker |
| `2'b11` | **SINGLE** | Single-flit packet вҖ” header + payload + tail combined |

### 3.3 Body/Tail Flit Layout

| Field | Bit Range | Width | Description |
|---|---|---|---|
| `flit_type` | `[511:510]` | 2 | `01` (Body) or `10` (Tail) |
| `vc_id` | `[509:508]` | 2 | VC for credit tracking |
| `packet_id` | `[507:500]` | 8 | Packet ID (matches header) |
| `payload` | `[499:0]` | 500 | Data payload (62.5 bytes effective) |

> **Note:** Body flits carry slightly less payload than the full 512 bits due to the 12-bit overhead for flit metadata. The effective per-flit data throughput is ~97.6%.

### 3.4 Opcode Encoding

| Opcode | Mnemonic | Direction | Description |
|---|---|---|---|
| `6'h00` | `RD_REQ` | Request (Plane 0) | Read request вҖ” address only, no data |
| `6'h01` | `WR_REQ` | Request (Plane 0) | Write request вҖ” address + data |
| `6'h02` | `ATOMIC_REQ` | Request (Plane 0) | Atomic operation (CAS, ADD, etc.) |
| `6'h10` | `RD_RESP` | Response (Plane 1) | Read response вҖ” data payload |
| `6'h11` | `WR_RESP` | Response (Plane 1) | Write acknowledgment (no data) |
| `6'h12` | `ATOMIC_RESP` | Response (Plane 1) | Atomic response вҖ” original data |
| `6'h20` | `BCAST` | Either plane | Broadcast вҖ” all nodes process |
| `6'h3F` | `NOP` | Either plane | No-operation / liveness probe |

### 4.2 Field Descriptions

| Field | Bits | Width | Description |
|---|---|---|---|
| `target_index` | `[55:48]` | 8 | **Target tile index** вҖ” encodes (x_dest, y_dest) as a flat index or hashed value. Used by the ATT (Address Translation Table) to look up NoC coordinates when the sender doesn't know the physical grid position. |
| `endpoint_id` | `[47:42]` | 6 | **Endpoint within tile** вҖ” selects which sub-block receives the transaction (e.g., L1 SRAM bank, register file, DMA, etc.). Up to 64 endpoints per tile. |
| `tlb_index` | `[41:36]` | 6 | **TLB/ATT entry index** вҖ” used by the `noc_routing_translation` logic to select the translation entry. Enables up to 64 translation contexts (e.g., per-process or per-VM). |
| `address` | `[35:0]` | 36 | **Local address** вҖ” byte address within the target endpoint. Supports up to 64 GB of address space per endpoint. |

### 5.2 VC Assignment

| VC | Purpose | Routing Modes | Priority |
|---|---|---|---|
| **VC 0** | Control / Register access | DIM_ORDER only | Low |
| **VC 1** | Bulk data (request) | DIM_ORDER, TENDRIL | Medium |
| **VC 2** | Adaptive data | DYNAMIC | Medium-High |
| **VC 3** | Response / Completion | All modes | **Highest** |

```

  в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ                   в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
```

## 6. Security Fence

### 6.1 `tt_noc_sec_fence_edc_wrapper`

### 6.3 SMN Group-Based Access Control

| Concept | Description |
|---|---|
| **SMN Group** | A logical grouping of source endpoints вҖ” e.g., "CPU cores", "DMA engines", "external host", "debug" |
| **Group assignment** | Derived from `(src_x, src_y, src_endpoint)` via a lookup table programmed at boot |
| **Rule table** | Per-tile array of rules: `{smn_group, allowed_opcodes, allowed_endpoints, address_range}` |
| **Default policy** | Deny-all вҖ” only explicitly allowed (group, opcode, endpoint) tuples pass |

### 6.4 Rule Matching

```systemverilog
// Pseudo-RTL for security fence check
wire [3:0] src_group = smn_group_table[{pkt.src_x, pkt.src_y, pkt.src_endpoint}];

wire rule_hit = |( smn_rules[i].valid &&
                   (smn_rules[i].group == src_group) &&
                   (smn_rules[i].opcode_mask & (1 << pkt.opcode)) &&
                   (smn_rules[i].endpoint_mask & (1 << pkt.endpoint_id)) &&
                   (pkt.address >= smn_rules[i].addr_base) &&
                   (pkt.address <  smn_rules[i].addr_limit)
                 ) for i in 0..NUM_RULES-1;

wire allow = rule_hit;
wire deny  = ~rule_hit;
```

### 6.5 EDC Integration

The `_edc_wrapper` suffix indicates that the security fence integrates an EDC1 node for:
- **Error detection** вҖ” parity checking on the firewall rule SRAM.
- **Error reporting** вҖ” `cor_err` flags propagated to the EDC ring for correctable errors in the rule table.
- **Error injection** вҖ” `err_inj_vec` from the EDC ring for manufacturing test of the security SRAM.

### 6.6 Security Violation Handling

| Event | Action |
|---|---|
| Unauthorized read | Return **error response** (AXI DECERR equivalent) on NoC; data is all-zeros |
| Unauthorized write | **Silently drop** write data; return error response |
| Any violation | Log `(src_group, src_x, src_y, opcode, address)` in security violation register |
| Any violation | Raise **security interrupt** to PLIC вҶ’ CPU firmware handles |

### 9.3 Repeater Architecture

```
tt_noc_repeaters_cardinal
в”ңв”Җв”Җ tt_noc_repeater_stage                // Pipeline register (flit data + valid)
в”Ӯ   в”ңв”Җв”Җ flit_reg [511:0]                // Full flit latched at noc_clk edge
в”Ӯ   в”ңв”Җв”Җ valid_reg                        // Flit valid signal
в”Ӯ   в””в”Җв”Җ vc_id_reg [1:0]                 // VC for credit tracking
в”Ӯ
в””в”Җв”Җ tt_noc_credit_relay                  // Credit forwarding
    в”ңв”Җв”Җ credit_in[3:0]                  // Credits from downstream (per VC)
    в”ңв”Җв”Җ credit_out[3:0]                 // Credits to upstream (per VC)
    в””в”Җв”Җ credit_reg[3:0]                 // 1-cycle credit pipeline
```

### 9.4 Properties

| Property | Value |
|---|---|
| Latency added | **1 cycle** per repeater stage |
| Data path | Full 512-bit flit + metadata registered |
| Credit path | 1-cycle credit relay (credit pipeline matches data pipeline) |
| Direction | **Cardinal** вҖ” NorthвҶ”South and EastвҶ”West (straight-line links only, no turns) |
| Transparency | Fully transparent to routing logic вҖ” routers see same credit semantics |
| Instance count | вүҘ5 (one per column at the repeater boundary, RTL: "multiple instances" of related modules) |

### 9.5 Impact on Routing Latency

| Path | Without Repeaters | With Repeaters | Delta |
|---|---|---|---|
| [3,0] вҶ’ [2,0] (same column, across boundary) | 1 hop = 1 cycle | 1 hop = **2 cycles** | +1 cycle |
| [3,0] вҶ’ [0,0] (3 hops vertical) | 3 cycles | **4 cycles** (1 repeater crossing) | +1 cycle |
| [3,0] вҶ’ [0,4] (3 hops vertical + 4 horizontal) | 7 cycles | **8 cycles** | +1 cycle |

The repeater adds latency but enables the NoC to run at higher `noc_clk` frequency, resulting in net throughput improvement.

## Appendix B вҖ” Glossary

| Term | Definition |
|---|---|
| **NoC** | Network-on-Chip вҖ” packet-switched on-die interconnect |
| **Flit** | Flow control unit вҖ” the smallest unit of transfer on the NoC (512 bits) |
| **VC** | Virtual Channel вҖ” logically independent buffer/queue sharing a physical link |
| **DOR** | Dimension-Ordered Routing вҖ” route in X first, then Y (deadlock-free) |
| **Tendril** | Source-routed with pre-computed waypoints (deterministic, load-balanced) |
| **Dynamic** | Adaptive routing вҖ” selects port based on real-time congestion |
| **NIU** | Network Interface Unit вҖ” bridges NoC to tile-internal AXI bus |
| **ATT** | Address Translation Table вҖ” maps addresses to NoC coordinates |
| **SMN** | System Maintenance Network вҖ” security/access control framework |
| **Cardinal** | Straight-line direction (N, S, E, W) вҖ” as opposed to diagonal |
| **Credit** | Flow-control token вҖ” sender must hold a credit to transmit a flit |
| **Arbiter tree** | Hierarchical round-robin arbitration structure |
| **Gasket** | Protocol/format conversion layer (here: AXI address вҶ” NoC address) |
| **BIST** | Built-In Self-Test |
| **SECDED** | Single Error Correct, Double Error Detect |
