## 1. Overview

Trinity N1B0 is a domain-specific AI accelerator SoC designed by Tenstorrent for high-throughput tensor computation. The chip uses a tiled architecture of heterogeneous compute elements interconnected by a packet-switched 2D mesh Network-on-Chip (NoC).

**Key Features:**

- 4×5 grid topology with 12 Tensix compute tiles, plus ARC, DRAM, ETH, PCI, and Router tiles
- Per-tile FPU + SFPU with dedicated register files (`tt_sfpu_lregs`) and instruction resource tracking (`tt_sfpu_instrn_resources_used`)
- TDMA engine with multi-channel DMA and thread context management (`tt_tdma_thread_context`)
- 2D mesh NoC with SECDED ECC, asynchronous CDC FIFOs, and cardinal repeaters
- NIU providing NoC-to-AXI bridge (`noc2axi_router_nw_opt`) with routing translation and self-test
- EDC1 serial ring bus with toggle-handshake protocol (`req_tgl`/`ack_tgl`), 4 modports (ingress/egress/edc_node/sram)
- SMN subsystem with clock divider (`tt_smn_clkdiv`) generating multiple clock domains from `noc_clk`
- Multi-domain clock architecture with `tt_clkbuf`, `tt_clkgater`, `tt_clk_gater`, `tt_clkdiv2` forming a distribution network
- DFX subsystem with memory wrappers and self-test capabilities

## 5. Compute Tile (Tensix)

### 5.1 SFPU (Scalar Floating-Point Unit)

**`tt_sfpu_lregs`** — Local register file (KB claims [6], [7]):
- Ports for register read/write operations, diagnostic data, error injection, and parity error reporting
- Implements register file with support for transposing or shifting data
- Reads and writes data/parity, reports parity errors

**`tt_sfpu_instrn_resources_used`** — Instruction resource tracker (KB claims [3], [8]):
- Analyzes SFPU instructions to determine register read/write hazards
- Tracks SFPU instruction status and stall conditions
- Multiple instances in the design

### 5.2 FPU (Floating-Point Unit)

- Performs fused multiply-accumulate (FMA) on tensor data
- Supports FP32, FP16, BF16, INT8 data types
- Reads from SRCB, writes to DEST register files

### 5.3 TDMA (Tensix DMA Engine)

**`tt_tdma_thread_context`** — Thread context manager (KB claim [5]):
- Generates addresses and control signals for multiple DMA channels
- Based on input instruction and configuration parameters
- Manages NoC ↔ L1 Cache data transfers

### 5.4 L1 Cache

- Per-tile SRAM scratchpad
- Banked architecture for simultaneous FPU/SFPU/DMA access
- Connected to EDC ring via `sram` modport for ECC protection

### 5.5 DEST / SRCB Registers

| Register | Purpose |
|----------|---------|
| DEST | FPU accumulation output |
| SRCB | FPU source operand B |
| SRCA | FPU source operand A (from L1) |

## 6. Dispatch Engine

### 6.1 East/West Structure

| Dispatch | Responsibility |
|----------|---------------|
| **East Dispatch** | FPU, SFPU, register file operations |
| **West Dispatch** | TDMA, L1 Cache, NoC I/F operations |

### 6.2 Command Distribution

```
         West NoC ──► [West Dispatch] ──┐
                                        ├──► Decode ──► FPU / SFPU / TDMA
         East NoC ──► [East Dispatch] ──┘
```

- Independent pipelines for parallel operation
- Opcode-based East/West routing from command queue

## 7. NoC Fabric

### 7.1 Routing Algorithms

| Algorithm | Description |
|-----------|-------------|
| **DOR (Dimension-Ordered)** | X-first → Y-next, deadlock-free |
| **Tendril** | Adaptive load-balancing with escape channels |
| **Dynamic** | Runtime-selectable per-packet routing |

### 7.2 Key NoC Modules (KB confirmed)

| Module | Function | Source |
|--------|----------|--------|
| `tt_noc_repeaters_cardinal` | Connects input/output NoC packages in cardinal directions | Claim [1] |
| `tt_noc_sync3_pulse` | 3-stage pulse synchronizer between clock domains | Claim [4] |
| `tt_noc_async_fifo_wr_side_reset` | Reset generation for async FIFO write/read sides | Claim [6] |

### 7.3 Flit Structure

- SECDED ECC protection (116-bit data + 10-bit check via `tt_noc_secded_chk_corr_116_10`)
- Flit types: HEAD, BODY, TAIL, SINGLE
- Credit-based flow control with virtual channels

## 8. NIU (Network Interface Unit)

**KB source:** NIU HDD section [13]

### 8.1 AXI Bridge

- **Module:** `noc2axi_router_nw_opt` — NoC-to-AXI router, northwest-optimized

- Self-test via `noc_routing_translation_selftest`

### 11.2 Ring Topology

```
┌────────┐    ┌────────┐    ┌────────┐         ┌────────┐
│ Tile 0 │───►│ Tile 1 │───►│ Tile 2 │───► … ──►│ Tile N │──┐
│(edc_   │◄───│(edc_   │◄───│(edc_   │◄── … ◄──│(edc_   │  │

└────────┘    └────────┘    └────────┘         └────────┘  │
     ▲                                                      │
     └──────────────────────────────────────────────────────┘
```

### 11.3 Serial Bus Protocol

- **Toggle-handshake**: `req_tgl` transition → node processes → `ack_tgl` transition
- No dedicated clock on serial link (asynchronous handshake)
- `ref_clk` domain for node-internal logic

### 11.4 Harvest Bypass

- Defective tiles bypassed via MUX: `ingress` → `egress` direct (skip `edc_node`)
- Ring integrity maintained regardless of harvest count
