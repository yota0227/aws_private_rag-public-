OPERATIONAL FLOW (Example: INT16 GEMM):

  1. Load operands:
     • SRCA[i] ← Matrix A row (via TRISC2 NoC read, ~40-50 cycles)
     • SRCB[j] ← Matrix B row (via TRISC2 NoC read, ~40-50 cycles)

  2. Enable FP-Lanes:
     • All 512 FP-Lanes start processing simultaneously
     • Each lane: A[col] × B[row] + DEST[k]

  3. Pipeline operation:
     • Cycle 0: Load operands into FP-Lanes
     • Cycles 1-8: Data flows through 8-stage pipeline
     • Cycle 8: First result available at DEST write
     • Cycle 9: Next result available (sustained 1/cycle)

  4. K-loop iteration:
     • While processing K partition, TRISC2 fetches next partition
     • Dual-bank SRCA/SRCB enables overlap (no stall)
     • Continue until K complete

  5. Output accumulation:
     • Results accumulate in DEST latch array
     • After GEMM complete, write DEST → L1 cache
     • Next operation begins

KEY INSIGHTS:

✓ FP-Lane is the fundamental execute unit
  └─ All 512 (256 per Tensix × 2 Tensix/cluster) work in parallel

✓ Format flexibility without additional hardware
  └─ Booth multiplier is format-agnostic (INT8/16/32, FP16B/32, etc.)
  └─ Same components, just reinterpreted bits

✓ Two-phase operation for INT8_2x mode
  └─ r0 and r1 lanes process different data in same cycle
  └─ Doubles INT8 throughput: 8 → 16 INT8 MACs/lane/cycle

✓ Zero-latency DEST integration
  └─ Latch arrays enable combinational read-modify-write
  └─ No pipeline stall for accumulation

✓ High aggregate throughput
  └─ 512 lanes × 8 MACs/lane/cycle = 4,096 INT16 MACs/cycle per Tensix
  └─ 8 Tensix × 4,096 = 32,768 INT16 MACs/cycle per cluster
