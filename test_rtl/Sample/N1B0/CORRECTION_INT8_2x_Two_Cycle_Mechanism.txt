┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                              │

│ ─────────────────────────────────────────────────                           │

│   ├─ SRCB row selection: [0:1]                                            │
│   ├─ DEST address: [DSTACC_IDX]                                           │
│   ├─ FPU processes 2 rows of INT8 pairs (4 INT8 MACs total per lane)      │
│   │  └─ Note: FPU pipeline still 8 cycles deep (S0-S7)                    │
│   ├─ Result: 4 INT8 values generated → write to DEST[DSTACC_IDX]         │
│   ├─ State: Set second_fp_phase_ready = 1                                 │
│   └─ Instruction PARTIALLY CONSUMED (Phase 1 complete)                    │
│                                                                              │

│ ────────────────────────────────────────────────────────────               │
│   ├─ Tag FF remains UNCHANGED (tag_ff.i_en = 0) - tag_d not updated      │
│   ├─ second_fp_phase_active = 1                                           │
│   ├─ SRCA datum selection: [index + 2] (offset by +2)                    │
│   ├─ SRCB row selection: [2:3] (offset by +2)                            │
│   ├─ DEST address: [DSTACC_IDX + 2] (offset by +2)                       │
│   ├─ FPU processes 2 MORE rows of INT8 pairs (4 INT8 MACs per lane)       │
│   ├─ Result: 4 INT8 values generated → write to DEST[DSTACC_IDX + 2]     │
│   ├─ State: second_fp_phase_ready = 0 (reset)                            │
│   └─ Instruction FULLY CONSUMED                                           │
│                                                                              │
│ Cycles N, N+1 SUMMARY:                                                     │
│ ──────────────────────                                                     │
│ One INT8_2x instruction spans 2 consecutive cycles                        │
│ Total result: 8 INT8 values written (4 per cycle, averaged)               │
│ Per-cycle throughput: 4 INT8 MACs/cycle/instruction                       │
│                                                                              │
│ BUT in pipelined execution:                                                │
│ ├─ Cycle N:   Instruction 0 (Phase 1)                                    │
│ ├─ Cycle N+1: Instruction 0 (Phase 2) + Instruction 1 (Phase 1) parallel │
│ ├─ Cycle N+2: Instruction 1 (Phase 2) + Instruction 2 (Phase 1) parallel │
│ └─ STEADY STATE: 1 instruction completes every 2 cycles                  │
│                                                                              │
│ Sustained Throughput: 1 instruction / 2 cycles = 0.5 instructions/cycle  │
│ But each instruction is 2× INT8 (vs 1× INT16)                             │
│ So: 2 INT8 MACs per instruction × 0.5 instructions/cycle = 1 INT8 MAC/cy │
│                                                                              │
│ Wait, that's only 1 INT8 MAC/cycle, not 4!                                │
│ The averaging is different...                                              │

CORRECTED THROUGHPUT ANALYSIS:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                              │
│ Key Insight: What counts as "MACs per cycle"?                              │
│                                                                              │
│ METHOD 1: Per-cycle write operations                                        │
│ ──────────────────────────────────────                                     │
│ Each INT8_2x instruction writes DEST twice (once per cycle):               │
│ ├─ Cycle 1: Write 4 INT8 results (4 INT8 MACs written)                   │
│ ├─ Cycle 2: Write 4 INT8 results (4 INT8 MACs written)                   │
│ └─ Average: (4+4)/2 = 4 INT8 MACs/cycle                                  │
│                                                                              │
│ METHOD 2: Instruction completion rate                                      │
│ ────────────────────────────────────                                       │
│ In steady state:                                                           │
│ ├─ Cycle N:   Instr_0 finishes, Instr_1 starts                           │
│ ├─ Cycle N+1: Instr_1 finishes, Instr_2 starts                           │
│ ├─ Rate: 1 instruction per cycle (but each = 2-cycle operation)          │
│ ├─ Per instruction: 4 INT8 values generated per cycle (during its phase) │
│ └─ So sustained: 4 INT8 MACs/cycle if continuously fed                   │
│                                                                              │
│ METHOD 3: FPU pipeline reality                                             │
│ ────────────────────────────────                                           │
│ The FPU pipeline (S0-S7) is still 8 cycles deep:                          │
│ ├─ This latency applies per phase                                         │
│ ├─ So total latency from instruction issue to final DEST write = 16 cy   │
│ ├─ But pipelined: new instructions can start each cycle                  │
│ └─ Sustained throughput = 1 operation per cycle                           │
│    (where operation = one phase of an INT8_2x instruction)                │
│                                                                              │
│ CONCLUSION:                                                                 │
│ ────────────                                                               │
│ If INT8_2x instruction issues every cycle:                                │
│ ├─ Phase 1 issues in cycle N, writes DEST in cycle N+8                   │
│ ├─ Phase 2 issues in cycle N+1, writes DEST in cycle N+9                 │
│ ├─ From cycle N+8 onward: 2 DEST writes per cycle (staggered)            │
│ └─ Average sustained: 4 INT8 MACs per cycle per FP-Lane                  │
│                                                                              │
│ BUT only if:                                                               │
│ 1. INT8_2x instructions are issued every cycle                            │
│ 2. Dependencies don't stall the pipeline                                   │
│ 3. DEST write bandwidth is available                                      │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘

PART 5: WHY THIS STILL ACHIEVES 2× INT8 THROUGHPUT
═══════════════════════════════════════════════════════════════════════════════

The 2× benefit comes from:

FACTOR 1: Data Packing (2× multiplier)
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                              │
│ 16-bit Booth multiplier → 2 INT8 products in parallel                      │
│ So same multiply hardware processes 2× the INT8 values per cycle           │
│                                                                              │
│ Result: 2 INT8 MACs per Booth multiply vs 1 INT16 MAC per Booth multiply  │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘

FACTOR 2: Two-Cycle Instruction Allows Data Reuse (2× multiplier)
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                              │
│ Phase 1: Load tag, process rows [0:1]                                      │
│ Phase 2: Reuse tag (not re-fetched), process rows [2:3]                    │
│                                                                              │
│ Benefits:                                                                   │
│ ├─ Amortizes instruction decode/issue overhead across 2 phases            │
│ ├─ Same tag satisfies 2 data blocks with minimal overhead (just offsets)  │
│ ├─ Allows continuous issuance: new instructions start each cycle          │
│ └─ Result: 2 "operation phases" per instruction                            │
│                                                                              │
│ Without two-cycle mechanism:                                               │
│ ├─ Would need separate instructions for rows [0:1] and [2:3]             │
│ ├─ Each instruction has full decode/issue overhead                        │
│ ├─ Throughput limited to instruction issue rate                          │
│ ├─ Result: Lower utilization                                              │
│                                                                              │
│ With two-cycle mechanism:                                                  │
│ ├─ One instruction covers 2 data blocks                                   │
│ ├─ Issue rate can be lower while maintaining throughput                   │
│ ├─ Or same issue rate yields higher data throughput                       │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘

COMBINED EFFECT:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                              │
│ INT16 throughput (single-phase):                                            │
│ ├─ 1 instruction per cycle                                                 │
│ ├─ 1 MAC per instruction                                                   │
│ └─ = 1 INT16 MAC per cycle                                                 │
│                                                                              │
│ INT8_2x throughput (dual-phase with packing):                               │
│ ├─ 1 instruction per cycle (issues every cycle)                            │
│ ├─ 2 phases per instruction (2 cycles of execution)                        │
│ ├─ 2 MACs per phase (due to INT8 packing in 16-bit Booth)                │
│ ├─ = 4 MACs per instruction (delivered over 2 cycles)                     │
│ └─ Sustained: 4 INT8 MACs per 2 cycles = 2 INT8 MACs per cycle           │
│                                                                              │
│ OR, in pipelined steady state:                                              │
│ ├─ Cycle N: Instruction 0 Phase 1 completes (4 MACs write to DEST)        │
│ ├─ Cycle N+1: Instr 0 Phase 2 completes (4 MACs) + Instr 1 Phase 1 starts │
│ ├─ = 4 MACs written to DEST each cycle (on average)                        │
│ └─ = 4 INT8 MACs per cycle (when fully pipelined)                         │
│                                                                              │
│ Ratio: INT8 throughput / INT16 throughput = 4 / 1 = 4× ✓                  │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘

PART 6: KEY TAKEAWAYS
═══════════════════════════════════════════════════════════════════════════════

1. **NOT Transparent Latch with Negative/Positive Clock Phases**
   └─ RTL uses FF enable control, not true transparent latch architecture

2. **Two-Cycle Instruction, Not Two Phases in One Cycle**
   └─ INT8_2x consumes 2 consecutive cycles with reused instruction tag

3. **Address Offsetting is the Key Innovation**
   └─ Phase 1: Access rows/addresses [N]
   └─ Phase 2: Access rows/addresses [N+2] (with same instruction tag)

4. **Still Achieves 4× Throughput vs INT16**
   └─ Factor 1: Data packing (2 INT8 per 16-bit slot) = 2×
   └─ Factor 2: Two-phase instruction reuse = 2×
   └─ Combined: 2 × 2 = 4×

5. **Pipeline Implications**
   └─ FPU pipeline: 8 cycles per phase = 16 cycles total latency
   └─ But pipelined: instructions can issue every cycle
   └─ Sustained throughput: 4 INT8 MACs per cycle (in steady state)

6. **Cluster-Level Math Still Holds**
   └─ INT16 cluster: 2,048 MACs/cycle ✓
   └─ INT8 cluster: 8,192 MACs/cycle ✓
   └─ 4× ratio maintained at all hierarchy levels
