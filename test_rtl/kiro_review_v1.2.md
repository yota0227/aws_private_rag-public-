# RTL HDD 자동 생성 현황 보고

**보고일:** 2026-04-22
**작성자:** 우승일

---

## 뭘 하고 있나

Trinity/N1B0 칩의 RTL 소스코드(9,465개 파일)를 RAG 시스템에 넣고, AI한테 "이 칩의 HDD 문서 만들어줘"라고 시키고 있습니다.

엔지니어가 수동으로 작성한 HDD를 정답지로 놓고, RAG가 자동 생성한 결과를 비교하면서 품질을 올리는 중입니다.

v3 R2 테스트(4/23 오후): max_results=20 Lambda 배포 후에도 5건만 반환 — 온프레미스 MCP Bridge가 파라미터를 안 넘기는 것으로 추정. 더 심각한 발견: HDD 86건이 전부 `trinity_noc2axi_router_ne_opt_FBLC` 단일 모듈에 편중. 5개 토픽 전부 같은 모듈. claim도 레지스터 래퍼(`_reg_inner`/`_wrap`)에서만 존재하고 데이터패스 모듈은 claim 없음. Claim 귀속 수정이 다음 단계의 핵심 블로커.

## 뭘 개선했길래 올라갔나

### v1 → v2 (30% → 40%)

처음에는 칩 전체를 한 문서로 뽑았는데, 너무 뭉뚱그려져서 토픽별(EDC, NoC, Overlay 등)로 6개 문서로 쪼갰습니다. EDC 보안 블록 228개 포트 분석 같은 건 오히려 엔지니어 문서보다 상세하게 나왔습니다.

### v2 → v2.5 (40% → 50%)

RAG 파이프라인에 심화 분석 5종을 추가했습니다:
- 패키지 파라미터 추출 (SizeX=4, SizeY=5, 타일 타입 enum 등)
- EDC 링 토폴로지 + 하베스트 바이패스
- NoC 라우팅 알고리즘 3종 + flit 구조
- Overlay CPU 클러스터 + L1 캐시 + APB 슬레이브
- SRAM 메모리 인스턴스 인벤토리

이 데이터들이 HDD 생성할 때 AI한테 같이 전달되니까 내용이 확 구체적으로 바뀌었습니다.

---

## 왜 아직 50%인가

AI가 못하는 게 아니라, RAG가 아직 RTL에서 뽑아주는 데이터가 부족합니다.

예를 들면:
- `i_ai_clk`이 1비트인지 4비트 배열인지 — RTL 파서가 비트폭을 안 뽑음
- `trinity_router`가 실제로 인스턴스화 되는지 안 되는지 — generate 블록 분석이 없음
- NoC 연결이 generate loop인지 manual assign인지 — assign 문 추적이 없음
- PRTN 데이지 체인 토폴로지 — 아예 토픽 분류에 없음

이런 정보가 RAG에 없으니까 AI가 "아마 이럴 거야"라고 추측해서 쓰는데, 그게 틀립니다.

---

## 핵심 포인트

AI 모델을 아무리 좋은 걸 써도, RAG가 정확한 데이터를 안 주면 품질이 안 올라갑니다.

| | AI 단독 | RAG + AI |
|---|---------|---------|
| 칩 파라미터 | 추측 (틀림) | pkg.sv에서 추출 (정확) |
| 모듈 계층 | 파일 있으면 쓴다고 판단 (틀림) | 인스턴스 관계 추출 (정확) |
| 토폴로지 | 일반 지식으로 생성 (대충 맞음) | 실제 RTL에서 분석 (정확) |

v1(30%)에서 v2.5(50%)로 올라간 20%p는 전부 RAG 파이프라인 개선 덕분입니다. 프롬프트 튜닝으로는 오류를 줄일 수 있지만, 커버리지 자체는 RAG 데이터가 없으면 못 올립니다.

## 결론

RAG Pipeline 품질 향상은 5월 중순까지 목표치(85% 이상)를 완수하기 위해 계속 집중할 것이며, 이는 RAG가 RTL 코드를 구조적으로 이해하고 설계 문서를 자동 생성함으로써, RTL 작성 → 검증 → 수정 과정에서의 SoC Test-Driven Development를 효과적으로 지원하는 기반 인프라가 될 것으로 판단합니다.
